# モデル差し替え（役割別レジストリ + Gradio プルダウン）計画

作成日: 2026-06-03 / 起案: ユーザー依頼（モデル差し替えを config 切替で統一し、Gradio に役割別プルダウンを追加）

---

## 0. 結論（ユーザー設問への回答）

> 「検出器 / トラッキング / 背景透過モデルを差し替えられる実装は、コンフィグ切替が一番良いのですよね」

**はい、コンフィグ切替が最適です。** 理由:

1. 既存コードが既にその設計。各重い Component が `config_name` / `checkpoint_path` /
   環境変数で差し替え可能で、SAMURAI は「config を samurai yaml に向けるだけ」で有効化される
   （`SAM2VideoPropagator.tracker_metadata()` 実装済み）。
2. SOLID/疎結合に合致。役割（detector / tracker / background）ごとに **同一 I/O 契約**の
   Haystack Component を用意すれば、選択は「どの設定を渡すか」だけになり分岐コードが増えない（YAGNI）。
3. バージョンアップ時の追加コストが最小。同一アーキの新 checkpoint は **レジストリ 1 行追加**、
   新アーキは Component 追加 + レジストリ 1 行で、Gradio プルダウンは自動で増える。

> 「Gradio UI に各役割のモデル切替プルダウンを作りたい」→ **合理的。** ただし下記の制約（§5）を
> 計画に織り込む。

## 1. 現状調査（確定事実）

| 役割 | Component | 切替インターフェース | 現状の選択肢 |
|------|-----------|----------------------|--------------|
| 検出器 (detector) | `GroundingDINODetector` / `GroundingDINOMultiBoxDetector` | `config_path` / `checkpoint_path`（env `GROUNDINGDINO_*`） | GroundingDINO SwinT OGC |

## 2. 設計方針（疎結合 / YAGNI）

### 2.0 前提（確定）

- **GPU 必須**: プロの現場で使う背景除去であり、CUDA GPU がある前提で設計する。
  重い推論（SAM2 / SAMURAI / GroundingDINO / transparent-background / MAM）は `require_gpu_for_heavy_inference`
  で CPU 暗黙実行を止める既存方針を踏襲する。CPU フォールバックは性能検証用途に限る。
  - **現状ギャップ（実装フェーズで要対応）**: `SAM2Segmenter.warm_up()` は既に `require_gpu_for_heavy_inference()`
    を呼ぶが、`TransparentBGExtractor` / `MAMAlphaPredictor` は未呼び出し。今回 transparent-background をベースにする
    以上、実装フェーズ（Phase R2 以降）で両者の先頭に `require_gpu_for_heavy_inference()` を追加する。
- **背景透過のベースは transparent-background**: 今回の開発は transparent-background を基軸に進める。
  MAM（`MAMAlphaPredictor`）は **将来の切替候補**として I/O 契約を今回定義し（§3.2）、アダプタ実装は次フェーズ。

### 2.1 単一の推論モデルレジストリ TOML を新設

`config/inference_models.toml`（新規）に役割別エントリを列挙する。Component の constructor 契約
（`config_name` / `checkpoint_path` / `device` / run 引数）はそのまま使い、新たな重い抽象は作らない。

確定スキーマ（field 定義）:

- `id`(必須): 一意キー。Dropdown の value。
- `label`(必須): Dropdown 表示名。
- `component`(必須): Component クラス名文字列。ローダがホワイトリスト（既知クラスのみ）で検証し、
  未知クラスは明示 raise（握りつぶさない）。
- `requires`(任意): 実行に必要なパッケージ変種の宣言。SAM2/SAMURAI は **同名パッケージの置換**のため
  1 環境にはどちらか一方しかインストールされない（§5.1 環境・インストール分離）。
  ローダは起動時に **どちらの変種がインストールされているか**を検出し、`requires` が一致する entry のみを
  可用とする。例 `requires = "sam2_facebook"` / `requires = "sam2_samurai"`。未指定は常時可用。
- 役割別 Component 引数（`config_path` / `checkpoint_path` / `config_name` / `tb_mode` 等）: そのまま
  callback が該当 Component の constructor / run へ渡す（§2.3 パターン A）。

```toml
[[detector]]
id = "groundingdino_swint_ogc"
label = "GroundingDINO SwinT-OGC (default)"
component = "GroundingDINODetector"
config_path = "GroundingDINO/groundingdino/config/GroundingDINO_SwinT_OGC.py"
checkpoint_path = "checkpoints/groundingdino_swint_ogc.pth"

[[tracker]]
id = "sam2_hiera_l"
label = "SAM2.1 Hiera-Large (default)"
component = "SAM2VideoPropagator"
config_name = "configs/sam2.1/sam2.1_hiera_l.yaml"
checkpoint_path = "checkpoints/SAM2/sam2.1_hiera_large.pt"
requires = "sam2_facebook"

[[tracker]]
id = "samurai_hiera_l"
label = "SAMURAI Hiera-Large (motion-aware)"
component = "SAM2VideoPropagator"
config_name = "configs/samurai/sam2.1_hiera_l.yaml"
checkpoint_path = "checkpoints/SAM2/sam2.1_hiera_large.pt"
requires = "sam2_samurai"

[[background]]
id = "tb_base"
label = "transparent-background base"
component = "TransparentBGExtractor"
tb_mode = "base"
```

> registry の版管理キーは現状不要（YAGNI）。将来必要になったら top-level `version` を追加する。

### 2.2 純粋なレジストリローダ（単一責務）

- `pipelines/components/model_registry.py`（新規）に **副作用なしの**ローダを置く:
  - `load_model_registry(path) -> dict[role, list[ModelEntry]]`（TOML 読込・検証のみ）
  - `entries_for(role) -> list[ModelEntry]` / `entry_by_id(role, id) -> ModelEntry`
  - `is_available(entry) -> bool`（`requires` と **インストールされている sam2 変種**を突き合わせ、§5.1）
    - 検出例: `requires` 未指定 / `sam2` 以外 → 常時 True。`sam2_*` は `import sam2` 成否と
      `samurai_mode` 設定（config / 環境変数）を突き合わせて判定する。SAMURAI もパッケージ名は `sam2`
      （同名フォーク）のため、単純な import 成否だけでは判別不可。インストール識別は環境変数または
      マーカファイル（例: 環境ごとに `INFERENCE_TRACKER_VARIANT=sam2_facebook|sam2_samurai`）で明示する。
  - `build_dropdown_choices(role) -> list[(label, id)]`（可用 entry のみ。純関数）
- ローダは TOML を読むだけ。**モデルは読まない**（import 時に checkpoint を触らない原則を維持）。

### 2.3 Gradio callback がレジストリ entry → Component を構成（ランタイム切替を明示設計）

現状 Gradio は `SAM2_PIPELINE = None` のグローバル単一キャッシュで Pipeline を一度だけ構築するため、
**そのままでは config 切替が効かない**（レビュー指摘 重要-1）。これを (role, entry_id) キャッシュへ拡張する。

- Gradio 側に役割別 `gr.Dropdown`（detector / tracker / background）を追加。choices は
  `build_dropdown_choices()` 由来（可用 entry のみ）、value は entry id。`info=` を完備。
- callback 実装は **パターン A（entry から引数を展開して constructor に渡す）**で固定:

```python
# 役割別の (entry_id -> 構築済み Pipeline) セッションキャッシュ
_PIPELINE_CACHE: dict[tuple[str, str], Pipeline] = {}

def get_sam2_pipeline(tracker_id: str) -> Pipeline:
    key = ("tracker", tracker_id)
    if key not in _PIPELINE_CACHE:
        entry = entry_by_id("tracker", tracker_id)          # registry 参照（純関数）
        propagator = SAM2VideoPropagator(                    # パターン A: 引数展開
            config_name=entry["config_name"],
            checkpoint_path=entry["checkpoint_path"],
        )
        _PIPELINE_CACHE[key] = build_sam2_tb_video_pipeline(propagator=propagator)
    return _PIPELINE_CACHE[key]
```

- callback は重い処理を一切持たず、`pipeline.run(..., include_outputs_from={...})` を**必ず明示指定**する
  （ERR018 再発防止）。
- 初回切替時のみ warm_up が走る（キャッシュ後は 0 コスト）。UI の `info=` に「切替時は再初期化が走る」旨を明記。

### 2.4 バージョンアップ運用フロー（ユーザー構想の明文化）

1. 新モデルが既存アーキと同一 → `inference_models.toml` に entry 1 行追加（プルダウン自動増）。
2. 新アーキ → 同一 I/O 契約の Haystack Component を追加（単機能・疎結合）→ entry 1 行追加。
3. checkpoint / config を `checkpoints/` と `config/` に配置 → ローダが検証。

## 3. I/O 契約（差し替え可能性の前提）

差し替えを成立させる鍵は **役割内で I/O 契約を固定する**こと。現状ほぼ満たすが計画で明文化:

### 3.1 役割別契約（現状）

- detector: 入力 `image, text_prompt` → 出力 `boxes`（および MaskSet 互換 dict）。
- tracker（静止画）: 入力 `image, points/box` → 出力 `mask_set`。tracker（動画）: 入力 `frames, points/box` →
  出力 `masks` dict + `tracker_metadata()`。
- background: 入力 `image, mask/selected_mask, モデル固有個別引数` → 出力 `rgba/alpha/preview/matte_result`
  （詳細は §3.2 統一 background 契約を参照）。

### 3.2 background ロール統一契約と MAM アダプタ（今回定義）

背景透過モデルを将来切替可能にするため、background ロールの **統一契約**を transparent-background の
現行出力に合わせて固定する。`MAMAlphaPredictor` はこの契約と入出力が異なるため、**アダプタで揃える**。

**統一 background 入力契約**（どの背景透過 Component もこれを受ける）:

| key | 型 | 必須 | 意味 |
|-----|-----|------|------|
| `image` | `np.ndarray` (HxWx3, uint8 RGB) | ✓ | 入力画像 |
| `mask` / `selected_mask` | `np.ndarray` (HxW) / `SelectedMask` dict | — | 対象領域。両方未指定は全体 |
| モデル固有引数（TB: `tb_mode/tb_jit/tb_threshold/tb_output_type/crop_padding`） | 個別引数 | — | モデル固有パラメータ |

> 【重要】モデル固有パラメータは Haystack Component の run() へ **個別引数**として渡す（`model_params` dict を
> Component 入力にはしない）。レジストリ entry の `[background]` フィールド群を callback の「パターン A: 引数展開」で
> run() の該当引数へ 1:1 対応させる。現行 `TransparentBGExtractor.run` はこの個別引数形式（既実装）。

**統一 background 出力契約**（`MatteResult` 安定 dict）:

| key | 型 | 意味 |
|-----|-----|------|
| `rgba` | `np.ndarray` (HxWx4, uint8) | RGBA 合成 |
| `alpha` | `np.ndarray` (HxW, uint8 0–255) | alpha matte |
| `preview` | `np.ndarray` (HxWx3, uint8) | 背景差し替えプレビュー |
| `matte_result` | `dict` | 上記 + `metadata{source, model_id, bbox, mask_used}` |

> 【現状ギャップ】現行 `TransparentBGExtractor` の `matte_result.metadata` に `model_id` は**未実装**
> （`source/tb_mode/bbox/mask_used` のみ）。マルチモデル切替で「どのモデルで処理したか」を追跡するため、
> 実装フェーズで `TransparentBGExtractor` / `MAMBackgroundAdapter` の constructor に `registry_entry_id: str | None = None`
> を追加し、run() で `metadata.model_id` に付与する（callback が registry entry id を constructor へ渡す）。

**`MAMAlphaPredictor` の現状 I/O（アダプタが橋渡しする対象）:**

- 入力: `image, task_type, box, points, point_labels, scribble_mode, guidance_mode`
- 出力: `alpha`（`np.ndarray` HxW, **float 0.0–1.0**）, `alpha_rgb`（`np.ndarray` HxWx3, uint8 グレースケール）

**アダプタ要件（`MAMBackgroundAdapter`、次フェーズ実装）:**

1. 統一入力→MAM 入力変換: `selected_mask`/`mask` から bbox を作り `task_type="text"`, `box=bbox` として MAM を呼ぶ
   （`model_params` で `guidance_mode` 等を受ける）。
2. MAM 出力→統一出力変換: `alpha`(float 0–1) を `alpha_u8 = clip(alpha*255).astype(uint8)` へ、
   `rgba = dstack([image, alpha_u8])`、`preview` は `compose_alpha`（TB と同じヘルパ）で生成。
   `matte_result.metadata.source = "mam"`。`model_id` は **constructor に渡される `registry_entry_id`** を使う
   （callback が entry id を constructor へ渡す。TransparentBGExtractor と同一方式）。
3. GPU 必須・`warm_up()` 遅延・`weights_only=True`（MAM は既に遵守）を維持。
4. アダプタは **純関数部分（出力変換）を `common.py` へ分離**し、unit test で GPU 不要検証できるようにする。

> 今回はこの契約定義まで。`MAMBackgroundAdapter` の実装と background Dropdown への MAM entry 追加は次フェーズ。
> 考慮点: MAM は `segment-anything` 依存、transparent-background は独立。同一環境で両立可（SAM2/SAMURAI と異なり名前衝突なし）。

新規 Component は上記契約に合わせる（`MaskSet` / `SelectedMask` / `MatteResult` 安定 dict）。

## 5. 制約・未決（正直な評価）

1. **SAM2 と SAMURAI の切替は環境・インストール分離を前提**: SAMURAI の `sam2` パッケージは本家 sam2 の
   置き換え（同名フォーク）。よって 1 プロセス内で「SAM2 ⇄ SAMURAI」をランタイム切替は**しない**。
   - **採用方針**: 切替は **環境・インストールの分離**で行う（例: SAM2 用 venv / SAMURAI 用 venv、
     または Colab セッション単位）。各環境には sam2 変種が 1 つだけインストールされる。
   - 起動時に `is_available()` が **その環境にインストールされた sam2 変種**を検出し、`requires` が一致する
     tracker entry のみを Dropdown choices に出す（他変種の entry は非表示）。つまり Dropdown は
     **同一変種内の checkpoint / config 違い**（hiera_l / hiera_t / mode）の切替を担う。
   - SAM2 ⇄ SAMURAI の切替を UI 上で見せたい場合は、起動時に検出された変種を UI 文言（`info=` /
     見出し）に明示し、「他方を使うには別環境で起動」と案内する。実装依存を増やさない（YAGNI）。
   - **実装中に万が一 sam2 と samurai が同居してエラーが顕在化したら** ERR0XX として `ERROR_LOG.md` に記録。
2. **ランタイム切替の warm_up コスト**: config 変更は predictor 再構築を伴う。(role, id) キャッシュで
   緩和するが、初回切替は再ロード待ちが発生。UI に「切替時は再初期化が走る」旨を `info=` で明示。
3. **VRAM**: 複数役割の同時常駐でピーク増。Text Prompt→動画の cache 解放方針（ERR030）を踏襲。
4. **MAM 系背景透過の統合**: `MAMAlphaPredictor` は I/O 契約が transparent-background と異なるため、
   `MAMBackgroundAdapter`（§3.2）で統一契約に揃える。**契約は今回定義済み**、アダプタ実装と
   background Dropdown への MAM entry 追加は次フェーズ。今回の background は transparent-background のみを choices に出す。

## 6. スコープ外（次フェーズ）

- 複数 box / マルチ obj union / 双方向伝播（別計画 `2026-06-02_...計画.md` の Phase A〜E）。
- MAM を background ロールへ統合する I/O アダプタ。
- レジストリのリモート（HF Hub 等）からの自動取得。
