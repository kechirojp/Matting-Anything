## 0. 背景・目的

「ドラムを叩いている男性」のような**複合対象（人＋ドラム）**を、動画全体で1被写体として
背景除去したい。現状は GroundingDINO の rank-1 bbox（最上位1個）のみを SAM2 に渡し、
1 obj_id で前方伝播しているため、以下の問題が出る。

1. 複合対象のうち1要素（例: 人だけ／帽子だけ）しか追跡されない。
2. 良いフレーム（ドラムが大きく写る等）を起点にできない（frame 0 固定）。
3. 動画途中フレームを起点にしても後方フレームが伝播されない（前方のみ）。
4. **アルファ下部が水平一直線で切れる**（後述の診断参照）。

本計画は「案3 UI（人手で候補選別）＋案2 内部（複数 obj_id 追跡→マスク union）」を軸に、
フレーム選択 UI（Q1）と双方向伝播（Q2）を追加する。

> **更新 (2026-06-03)**: SAMURAI リポジトリが `samurai/` にクローン済みのため、本計画に
> **SAMURAI 統合（セクション 9）** と **トラッキング可視化 UI（セクション 10）** を追加した。

## 1. 現状調査（確定事実）

- `detect_text_boxes_for_video`：`boxes[0]`（最上位1個）のみ `prompt_state["box"]` にコピー。
  残り候補は Dataframe 表示のみ。
- `prompt_state`（`ui_helpers.empty_prompt_state`）：単一 `box` 前提
  （`points/labels/box/box_buffer/input_mode`）。
- `SAM2VideoPropagator.run`（`video_model_components.py` L186-）：
  `box`/`points` + `object_id=1` の単一物体。`add_new_points_or_box(frame_idx=0)` 固定。
  `propagate_in_video(state)` は**前方のみ**。propagate ループは全 obj_id を返すが
  `object_id` 1個分しか拾っていない。
- `TransparentBGExtractor.run`（`model_components.py` L609-）：
  frame ごとに SAM2 mask の bbox（`mask_to_bbox(mask, padding=crop_padding=40)`）で **crop** し、
  crop 外の `full_alpha` を **0 埋め**。
- `common.py` に `union_masks(mode="or", dilate_kernel)` 既存（静止画版で実績）。
- `VideoReader` は `metadata["sampled_frame_indices"]` に実フレーム番号を保持。

## 2. アルファ下部が水平一直線で切れる問題の診断

### 症状
30 フレーム後のマスク下部が**水平直線**でカットされる（本来は画像下端までアルファが必要）。
動画: ドラムを叩く男性、カメラが右 2:45 方向へ徐々に引いていく、カメラ高さ約 1m。

### 根本原因
`TransparentBGExtractor.run` が **毎フレーム** SAM2 mask の bbox で crop し、
crop 外を `full_alpha[y_min:y_max, x_min:x_max] = alpha_crop` で 0 埋めする。
→ その frame の **SAM2 mask の下端（+padding）が水平カット線**になり、その下では
transparent-background が走らず alpha=0 になる。

### ユーザー仮説「BBOX が固定のまま」は誤り
bbox は固定ではなく、`mask_to_bbox` で**毎フレーム再計算**されている。
真因は **SAM2 伝搬マスクの下部が 30 フレームで後退・欠落**していること。
その結果、毎フレーム算出される bbox 下端が上方へ移動し、水平カット線として現れる。

### SAM2 マスク下部欠落の推定理由
- frame 0 で単一 box prompt（rank-1 DINO）を 1 obj_id 登録。
- 手前のドラム／脚など**別オブジェクト**は追跡対象外。
- カメラ引き＋被写体の動きで、下部領域の confidence が低下しマスクが上方へ後退。

### 対策
| 案 | 内容 | 評価 |
|----|------|------|
| **(a)** | 複数 obj union（本計画 Phase B）。下部のドラム/脚を別 obj として追跡し union | ◎ 本筋 |
| (b) | crop の hard-zero を廃し画像全体で transparent-background | △ 性能トレードオフ |
| (c) | `crop_padding` 拡大 | × 対症療法。水平線位置が下がるだけ |

→ 本質は SAM2 マスクの下部欠落。**(a) 複数 obj union が本筋**。

## 3. Q1: 手動シークで「DINO が箱を作りやすいフレーム」を選ぶ

### 結論
`gr.Video` の再生シーク位置（`currentTime`）は **Python backend に渡らない**（Gradio 仕様）。
→ ユーザー推測「何フレーム目か取得できない」は**正しい**。

### 解決策（フレーム番号スライダー）
- `gr.Slider(0, frame_count-1, step=1)` でフレーム番号を指定。
- `cv2.VideoCapture` + `cap.set(cv2.CAP_PROP_POS_FRAMES, n)` で当該フレームを抽出しプレビュー。
- そのフレームに DINO 検出をかける → **frame_idx が確実に取得できる**。
- 素地あり：`VideoReader` の `sampled_frame_indices`、現状 `extract_first_frame` は frame 0 決め打ち。
- ERR017（座標手入力禁止）には**非抵触**（フレーム選択であって bbox 座標入力ではない）。

## 4. Q2: SAM2 の双方向伝播

### 結論
SAM2 は双方向伝播**可能**。`propagate_in_video(state, reverse=True)` で後方伝播。
初期化フレーム `frame_idx=N` に box 登録後、`reverse=False`／`reverse=True` を 2 回回せば
**全フレームをカバー**できる（途中フレームを起点にしても前後とも埋まる）。

現状は `add_new_points_or_box(frame_idx=0)` 固定・前方のみ。
→「**初期化フレーム可変＋双方向**」をセットで実装する必要がある。
SAMURAI も SAM2 ベースのため同 API を想定。

## 5. 実装 Steps

### Phase A: prompt_state 複数 box 対応（基盤）
1. `ui_helpers.empty_prompt_state()` に `"boxes": []` を追加。`copy_prompt_state()` で深いコピー。
2. `draw_prompt_overlay()` で `boxes` を複数矩形（色変え／番号付与）として描画。単一 `box` 描画は維持。

### Phase B: SAM2 マルチ obj 追跡 + union + 初期化フレーム可変 + 双方向（コア）
3. `SAM2VideoPropagator.run` に `boxes: list[list[int]] | None`、`prompt_frame_idx: int = 0`、
   `bidirectional: bool = False` を追加。
   - boxes 指定時：obj_id 1..N で各 box を `add_new_points_or_box(frame_idx=prompt_frame_idx, ...)`。
   - propagate ループで frame ごとに全 obj mask を OR して `frame_masks[source_index]=union`。
   - `bidirectional` 時：`reverse=False` と `reverse=True` の 2 パスを回す。
   - 未指定時：既存の単一 box/points/object_id・frame_idx=0・前方のみを維持（後方互換）。
4. メモリ/性能：複数 obj は `propagate_in_video` 1 回で同時伝搬。`torch.cuda.empty_cache()` は既存どおり。

### Phase C: Pipeline 配線
5. `pipelines/sam2_tb_video_pipeline.py` は socket 不変のため結線変更なし。run 引数追加のみ確認。

### Phase D: Gradio UI（案3 入口 + フレーム選択）
6. **フレーム選択 UI**：`gr.Slider`（フレーム番号）＋「このフレームを表示」ボタン。
   `cv2` で当該フレームを抽出し Prompt Canvas に表示。選んだ `prompt_frame_idx` を state に保持。
7. `detect_text_boxes_for_video` 改修：選択フレームに DINO → 候補を `gr.CheckboxGroup` で提示
   （top1 既定チェック）。Dataframe は表示用に維持。
8. 新規 callback `apply_selected_boxes`：選択候補を `prompt_state["boxes"]` にコピー、overlay 更新。
9. `run_video_background_removal`：`boxes`／`prompt_frame_idx`／`bidirectional` を propagator に渡す。
   バリデーションを points/box/boxes のいずれか必須に更新。
10. UI 文言・`info=` 完備（Slider／CheckboxGroup／双方向チェック／各ボタン）。

### Phase E: テスト・検証
11. `tests/unit/test_video_pipeline_wiring.py`：`boxes`／`prompt_frame_idx`／`bidirectional` パラメータ存在 assert。
    fake predictor で 2 obj mask の OR と reverse 2 パスを検証。
12. `ui_helpers` の boxes コピー／描画 unit test（`copy_prompt_state` が boxes を共有参照しない）。
13. Notebook 正本 `Sam2_Transparent_Background_Haystack_for_Movie.py` は UI 起動のみ。
    文言/導線変更が及ぶ場合のみ Jupytext 再生成。

## 9. SAMURAI 統合（追加：2026-06-03）

### 9.1 SAMURAI とは
- `samurai/sam2/` に置かれた **SAM2 のフォーク**。SAM2 本体に Kalman filter（motion-aware
  memory）を追加し、オクルージョン・高速動体の追跡安定性を高める。
- **有効化は config の `samurai_mode: true` のみ**（`samurai/sam2/sam2/configs/samurai/sam2.1_hiera_*.yaml`）。
  コード API は SAM2 と**完全に同一**（`build_sam2_video_predictor` / `init_state` /
  `add_new_points_or_box` / `propagate_in_video`）。
- 依存: SAM 2.1 checkpoint + `pip install -e samurai/sam2` + `scipy loguru`。`python>=3.10`, `torch>=2.3.1`。
- SAMURAI の `sam2` パッケージは本家 sam2 の**置き換え**（同名フォーク）。両立不可。

### 9.2 統合方針（コード変更を最小化）
- `SAM2VideoPropagator` は既に `SAM2_CONFIG_NAME` / `SAM2_CKPT_PATH` を env から読むため、
  **config を samurai のものに向ける**だけで SAMURAI 化の大半が可能。
- `SAM2_CONFIG_NAME` の既定を `configs/samurai/sam2.1_hiera_l.yaml`、`SAM2_CKPT_PATH` の
  既定を SAM 2.1 large checkpoint に切り替え可能にする（env 未設定時の fallback）。
- 互換性: env で従来 sam2 config を指定すれば旧動作に戻せる。`samurai_mode` は config 依存のため
  コード側分岐不要。
- Component は `samurai_mode` の有無を `masks` metadata に記録し、可視化 UI で表示できるようにする。

### 9.3 環境セットアップ（ユーザー確認項）
- 現行 `segment-anything` / `sam2` と SAMURAI 版 sam2 の共存方針を決定する（本計画の未決 5）。
- 検証は GPU + checkpoint が必要なため `@pytest.mark.integration` 骨格にとどめ、実推論は Colab/GPU で確認。

## 10. トラッキング可視化 UI（追加：2026-06-03、ユーザー明示依頼）

### 10.1 目的
現状、SAM2/SAMURAI の mask が各 frame で**正しく追従しているかを確認する手段がない**。
背景除去結果だけでは追跡失敗（マスク後退・乗り移り）に気付けないため、**mask overlay プレビュー動画**を出す。

### 10.2 設計（責務分担：Component に委譲）
- 新規 Component `TrackingOverlayWriter`（`video_model_components.py`）を追加。
  - 入力: `frames`、`masks`（frame_masks）、`metadata`、`progress_callback`。
  - 処理: frame ごとに mask 輪郭と半透明塗りつぶし（obj_id ごとに色分け）を元画像に重ね、
    `outputs/{timestamp}/video/tracking_overlay.mp4` と `sequence/overlay/` に**逐次保存**（ERR030: RAM に全 frame を保持しない）。
  - 出力: `overlay_video_path`。
- mask 描画は `common.py` の既存ユーティリティ（`compose_alpha` 等）を流用し、重複実装しない。
- Gradio callback は overlay 動画 path を `gr.Video` に表示するのみ（重い描画は Component に委譲）。

### 10.3 UI 要素
- 出力タブに `gr.Video`「Tracking Overlay（追跡確認用）」を追加。背景除去出力と並列で並べ、
  mask が被写体に追従しているかを目視確認できるようにする。
- `info=` で「色付き領域＝SAM2/SAMURAI が追跡している mask。被写体からずれていたら追跡失敗」と説明。
- 可視化の ON/OFF を `gr.Checkbox`（既定 ON）で切替（overlay 生成は追加コストのため）。

### 10.4 検証
- `tests/unit/test_video_pipeline_wiring.py`：`TrackingOverlayWriter` の存在と、fake frames/masks で
  overlay frame が元画サイズと一致し色が乗ることを検証（integration 不要の軽量描画テスト）。
- `.github/skills/webapp-testing` + `ui-ux-pro-max`：Gradio 実起動 → Playwright で Tracking Overlay タブと導線をスクショ。
