## 実装方針

### 推奨フロー

```
1. RF-DETRで毎フレーム or NフレームごとにBBOX検出
2. ByteTrack/BoT-SORTでtrack_idを確定
3. track_idごとのBBOXをSAM2.1へprompt
4. SAM出力maskをtrack_id単位で保存
5. maskが破綻したら、BBOX検出側で再プロンプト
6. alpha生成はBiRefNet系で細部を補正
7. temporal smoothing / optical flow / confidence blendでちらつき抑制
```

### なぜこの構成がよいか

- **ID管理はMOTが得意**
- **境界抽出はSAM/BiRefNetが得意**
- **alphaの髪・半透明はbinary maskだけでは無理**
- **SAM2.1のtracking driftはBBOX再投入で補正できる**
- **商用ライセンス地雷を避けやすい**

## ステージ別・ライセンス確定版

### ① 検出（YOLO代替の本命＝RF-DETR）

YOLO系を避ける理由が裏取りで明確になりました。Ultralytics YOLOの学習済みモデルはすべてデフォルトでAGPL-3.0で、AGPL-3.0は学習コードとそのコードが生成したモデルの両方をカバーします [\[2\]](https://www.ultralytics.com/license) 。AGPL-3.0準拠は、より大きなアプリ・改変・スクリプト・設定ファイル、場合によってはモデル重みを含む派生物全体のソースコード公開を意味し、フルプロジェクトをオープンソースにしたくないならEnterprise Licenseが必要 [\[2\]](https://www.ultralytics.com/license) です。YOLO26も同様で、YOLO26のコード・モデル・ドキュメントはAGPL-3.0およびEnterpriseライセンス下 [\[4\]](https://docs.ultralytics.com/models/yolo26) です。

対してRF-DETRは商用に明確に安全です。オープンソースのrfdetrパッケージとApache指定のモデル重みはApache 2.0ライセンスで、Plusコンポーネント（rfdetr\_plus、RF-DETR-XL/2XL検出モデル）はPML 1.0 [\[1\]](https://github.com/roboflow/rf-detr) 。つまり **N/S/M/LはApache 2.0、XL/2XLだけPML 1.0で要注意** という前回の認識は正しかったです。最新性も確認でき、2026年1月にRF-DETR v1.4と同時にRF-DETR-Segが登場し、全チェックポイント（Nano〜2XLarge）がApache 2.0で、ソースコード開示不要で商用利用可能 [\[2\]](https://learnopencv.com/rf-detr-segmentation-real-time-detection-instance-segmentation-guide/) 。性能面もRF-DETR-LはT4で6.8msのレイテンシで56.5 AP50:95を達成し、YOLOv11x（54.7 AP）をより低レイテンシで上回ります [\[3\]](https://rfdetr.roboflow.com/develop/) 。

### ② 複数トラッキング（ここはMOT専用ライブラリに任せる）

前回どおり、ID管理はSAMでなくMOT側が正解です。Roboflow trackersが商用安全で最適です。Roboflow trackersは主要な多物体追跡アルゴリズムをApache 2.0でクリーンに再実装したもので、既存の任意の検出モデルと組み合わせられます [\[1\]](https://github.com/roboflow/trackers) 。SORT・ByteTrack・OC-SORT・BoT-SORTのベンチ済み実装を提供し、YOLO・DETR・RT-DETRなど任意の検出器にグルーコードなしで差し込めます [\[1\]](https://github.com/roboflow/trackers) 。

選び方:

- **ByteTrack**: 箱出しで4ベンチ中2つで最良、変動する確信度の検出器をうまく扱い、リアルタイムレイテンシを提供 [\[3\]](https://trackers.roboflow.com/latest/) 。あなたの高速性重視（10秒に30分は論外）に合致。
- **BoT-SORT**: 遮蔽・カメラ移動向け。ByteTrackをカメラ移動補正（CMC）で拡張し、二段階アソシエーションを保ちつつ光学フロー由来の幾何変換を先に適用してカメラのエゴモーションによる物体のジャンプに対処 [\[8\]](https://trackers.roboflow.com/develop/trackers/botsort/) 。

### ③ SAM2.1（マスク生成）── ここは完全に安全と確定

前回「2.1重みに別条件がないか要確認」とした点、クリアです。SAM 2のモデルチェックポイント、デモコード（フロント/バック）、学習コードはApache 2.0でライセンスされています [\[5\]](https://github.com/yangchris11/samurai/blob/master/sam2/README.md) （SAMURAIリポジトリ内のSAM2も同じ）。Meta はSAM 2をApache 2.0で公開し、学習に使ったSA-VデータセットはCC BY 4.0で共有 [\[3\]](https://ai.meta.com/blog/segment-anything-2/) 。つまりコードも重みもApache 2.0で、商用組み込みに地雷なしです。

SAM2.1のトラッキング補強としてSAMURAIも安全圏です。SAM2は物体セグメンテーションは強いが、混雑シーンや高速・自己遮蔽する物体での視覚追跡に弱く、固定窓メモリが次フレームの条件付けに使うメモリ品質を考慮せず誤差伝播を招く。SAMURAIはこの追跡用にSAM2を強化した適応版 [\[3\]](https://yangchris11.github.io/samurai/) です。ただし **SAMURAIはVOT（単一物体追跡）寄り** なので、複数同時トラッキング要件にはやはり「RF-DETR＋トラッカー＋SAM2.1」のほうが堅い、という前回の判断は維持します。

### ④ 動画マッティング（一番痩せた領域＝裏取りで確定）

ここが最重要の裏取りでした。 **MatAnyoneは非商用で確定** です。MatAnyone 2はNTU S-Lab License 1.0でライセンスされています [\[2\]](https://github.com/pq-yang/MatAnyone2) 。そしてS-Lab License 1.0は「非商用目的（non-commercial purpose）」での再配布・使用のみ許可 [\[1\]](https://github.com/pq-yang/MatAnyone/blob/main/LICENSE) 。性能は魅力的でCRGNN実世界ベンチでMatAnyone 2はMatAnyone 1比でMADを−26%、勾配誤差を−24.5%削減 [\[4\]](https://gaga.art/blog/matanyone-2/) ですが、 **商用OSS縛りでは採用不可** です。一部のSaaSが「商用OK」と書いていますが、それは自社ホスト型サービスの利用規約の話であって、モデル重みの自社プロダクト組み込みとは別問題なので混同注意です。

→ 商用安全で固めるなら **BiRefNet** が現実解。モデル重みはMITライセンスで提供 [\[2\]](https://docs.comfy.org/tutorials/utility/remove-background-birefnet) 、License: mit [\[1\]](https://huggingface.co/ZhengPeng7/BiRefNet) でコードもHF重みもMIT。高解像度・自動・単一マスクの前景抽出を、ユーザー入力なし＆実際に使える商用ライセンスで欲しいときのスイートスポット [\[5\]](https://ice-ice-bear.github.io/posts/2026-04-15-birefnet/) 。ただし **画像/フレーム単位なので動画の時間方向のちらつきは自前で抑える** 必要があります。速度の参考: RTX A5000で1024×1024が1秒未満（CPUでは秒単位で厳しい） [\[5\]](https://ice-ice-bear.github.io/posts/2026-04-15-birefnet/) 。

#### SAM3はどうか（前回保留分の確定）

SAM3は「OSS + 商用可」の厳密な定義からは外れます。形式的にはApacheではなくMeta独自のSAMライセンスで、研究と商用の幅広い利用を許すが軍事・ITAR用途を禁止 [\[1\]](https://sam3ai.com/license/) 。OSI（Open Source Initiative）は通常こうした「利用分野の制限」を厳密なオープンソース定義とは非互換とみなします [\[1\]](https://sam3ai.com/license/) 。さらに再配布時（顧客に重みを出荷する場合など）はSAM3自体をSAMライセンス下に保つ必要 [\[1\]](https://sam3ai.com/license/) があり、重みもgated配布です。実用上は使えますが、 **「純粋OSS」を要件にするなら採用は保留が妥当** です。

## 結論：BEN2 base は「商用OSS + 動画標準対応」で本命候補

前回のver1.2でBiRefNetを④に置きましたが、BEN2 baseのほうが **動画パイプラインを自前で組まなくて済む** ぶん優位です。ライセンスも問題ありません。

### ライセンス確定

HuggingFaceのモデルカード／リポジトリ両方で確認でき、BEN2はlicense: mitで配布されています [\[3\]](https://huggingface.co/PramaLLC/BEN2/blob/main/README.md) 。さらに開発元PramaLLCが明言しており、「BENモデルは問題なく商用利用できる。Apache 2.0ライセンス下にあり、商用ピースはBEN+Refinerのみで、BEN\_BASEは商用利用に全く問題ない」 [\[3\]](https://github.com/DoctorDiffusion/ComfyUI-BEN) とのことです。

※ここで注意点（review²向けの正確性）：HuggingFaceのタグは「MIT」、開発元の発言は「Apache 2.0」と **表記が食い違っています** 。MITもApache 2.0もどちらも商用可・コピーレフトなしのパーミッシブライセンスなので **実務上の結論（商用OK）は変わりません** が、厳密に裏取りするなら、実際にダウンロードする重みファイルに同梱されるLICENSEファイル本文を確認すべきです。タグ＝MITが現行の正式表記と見るのが妥当です。

そして **有償なのはRefiner（BEN+Refiner / 商用フルモデル）だけ** で、これは前回のMatAnyone/SAM3とは違い「baseだけで完結して商用OSSで使える」点が決定的に良い。Base modelはオープンソースで、フルモデルは無料Webデモまたは商用API（ [sales@backgrounderase.com](mailto:sales@backgrounderase.com) ）経由 [\[2\]](https://huggingface.co/PramaLLC/BEN2) という切り分けです。

### 動画対応＝標準実装（ここが対BiRefNetの最大の利点）

BiRefNetは画像/フレーム単位で時間方向のちらつきを自前で抑える必要がありましたが、BEN2は動画関数が組み込み済みです。BEN2は画像処理に加えて動画セグメンテーション機能を提供し、動画パスを指定するだけで自動処理してforeground.webmまたはforeground.mp4として保存できます [\[4\]](https://www.aibase.com/news/15177) 。

API設計も実用的で、 `segment_video()` に以下が揃っています：fps（0で元動画のfpsを自動検出）、refine\_foreground（エッジ改善の追加後処理）、webm=Trueでアルファレイヤー動画を出力、webm=Falseの場合はrgb\_valueで背景色を指定（デフォルトは緑(0,255,0)） [\[2\]](https://huggingface.co/PramaLLC/BEN2) 。アルファ出力（webm）が標準で出せるので、透明背景への受け渡し要件に直結します。

### 品質・速度

精度面は強い。BEN2はDIS5kと22K独自セグメンテーションデータセットで学習され、髪のマッティング、4K処理、物体セグメンテーション、エッジ精細化で優れた性能を発揮します [\[2\]](https://huggingface.co/PramaLLC/BEN2) 。あなたが重視する髪の毛エッジに効くCGM構造です。BEN2はConfidence Guided Matting (CGM)パイプラインを用い、ベースモデルが低信頼とした画素を専用リファインネットワークで処理して、より精密なマッティングを実現します [\[4\]](https://www.aibase.com/news/15177) 。

ただし速度の但し書き：上記の高精度なエッジ処理（refine\_foreground）はbaseでも使えますが、refine\_foregroundはエッジを改善する反面、推論時間が増える追加後処理で、デフォルトはFalse [\[6\]](https://github.com/PramaLLC/BEN2/) です。コンシューマGPup運用なら、バッチサイズは3を超えないことが推奨（それ以上は推論ゲインがほぼない） [\[6\]](https://github.com/PramaLLC/BEN2/) 。ONNX版（onnx-community/BEN2-ONNX、これもMIT）もあるので軽量デプロイの選択肢になります。

```
RF-DETR (N/S/M/L, Apache 2.0)
  ↓ BBOX
Roboflow trackers ByteTrack (Apache 2.0)   ← 高速重視
  ↓ track_id付きBBOX
SAM2.1 (Apache 2.0)         ← 複数ID個別マスクが要るならここ
  ↓ binary mask
BEN2 base (MIT) ← 動画matting標準対応 / webmでアルファ直接出力
```

ただし設計上の重要分岐があります。 **BEN2 baseは「画面全体の前景/背景」を分離するモデルで、SAM2.1のような「BBOXやポイント指定で特定オブジェクトだけ」を選ぶ仕組みは持ちません** （検索結果上、prompt入力APIは確認できず、入力は画像/動画パスのみ）。つまり：

- **用途A：画面の前景まるごとを透明背景化** → 検出もトラッカーもSAM2.1も不要で、 **BEN2 base単体（ `segment_video` ）で完結** 。これが一番軽くてシンプル。
- **用途B：複数人物を個別IDで切り出す／特定対象だけ抜く** → BEN2では対象選択できないので、 **SAM2.1（プロンプト可・複数ID）が必須** で、BEN2はその後段の「エッジ仕上げ」用途に限定されるか、そもそも不要になる。

あなたの「複数BBOX/複数ポイントのマスク統合→アルファ化」という過去の検討文脈からすると、 **用途B寄り＝SAM2.1が主役、BEN2は単体運用なら用途A** という整理になります。

理由は、 **BEN2はマスク誘導モデルではなく、入力画像/動画から前景を推定する背景除去モデル** なので、SAMの二値シルエットやinpaint済み外側を見せると、むしろ不自然な境界・生成ノイズ・フレーム間ちらつきを追加する可能性が高いです。BEN2公式APIも `inference(image)` / `segment_video(video_path, ...)` で、任意マスクをプロンプトとして渡す口は確認できません。([github.com](https://github.com/PramaLLC/BEN2))

## 推奨パイプライン

```
1. BEN2単体で全体前景マット取得
   ├─ A_ben_full: BEN2の素アルファ
   └─ BEN2結果プレビュー

2. BBOX検出 + 手入力補完

3. SAM2.1で対象マスク作成
   ├─ box prompt
   ├─ positive points
   ├─ negative points
   └─ 可能なら binary だけでなく logits/probability も保存

4. SAMマスクは「正解シルエット/制約マスク」として使う

5. 最終アルファは
   SAMの内外制約 + BEN2の境界アルファ
   で合成
```

SAM2は動画内の任意フレームに対して、クリック、BBOX、マスクで対象を指定・修正し、その結果を動画全体へ伝播できます。追加プロンプトで後から修正できる設計なので、あなたの「検出できないものを手入力」「ポジ/ネガポイント追加」には合っています。([ai.meta.com](https://ai.meta.com/blog/segment-anything-2/))

## 重要：SAMの二値マスクは「アルファ」ではなく「制約」にする

あなたの認識どおり、SAM2.1の二値輪郭はそのまま透明化には荒いです。  
ただし、 **使えないわけではない** です。

使い方はこれ。

```
SAM mask = 対象がどこにいるかの正解領域
BEN2 alpha = 境界・髪・ぼかし・半透明の候補
```

つまり最終合成はこう。

```
M_sam = SAMの対象マスク
A_ben = BEN2のアルファ

core    = erode(M_sam)
outside = not dilate(M_sam)
band    = dilate(M_sam) - erode(M_sam)

A_final:
  core    → 1.0寄りに固定
  outside → 0.0に固定
  band    → BEN2 alphaを採用
```

イメージ：

```
SAM内側の確実な領域      → 不透明
SAM外側の確実な背景      → 透明
SAM境界の数px〜数十px帯  → BEN2に任せる
```

これなら、SAMの二値境界をそのまま出さずに、BEN2のマッティング能力を境界だけに使えます。

## 「inpaintしてBEN2」は基本NG

あなたの案：

```
SAMシルエット外をinpaint
↓
BEN2で再処理
```

これは一見よさそうだけど、リスクがあります。

### NG寄りの理由

- BEN2は「自然画像内の前景/背景関係」を見て前景を抜くモデル。
- inpaint外側は生成された偽背景なので、元動画との照明・テクスチャ・境界がズレる。
- 動画だとinpaint結果がフレームごとに揺れて、BEN2入力自体が不安定になる。
- SAM境界が硬いまま残ると、BEN2にとって「人工的な輪郭ノイズ」になる。

なので、 **inpaintはBEN2の前ではなく後** が安全です。

```
透明化したい場合：
  original + A_final → RGBA

背景も自然に埋めたい場合：
  original + object mask → video inpaint
  ただしこれは透明化とは別処理
```

## 例外：BEN2をもう一度使うなら「inpaint」ではなく「crop」

もしBEN2全体処理で対象を取りこぼすなら、再処理はこう。

```
SAM/BBOXから対象ごとの拡張cropを作る
↓
元画像のcropをそのままBEN2へ入れる
↓
crop内のBEN2 alphaを取得
↓
SAM maskの境界bandだけに貼り戻す
```

ポイントは、 **元画像を壊さずにcropする** こと。

```
良い：
  original frame の bbox拡張crop → BEN2

悪い：
  SAM外をinpaint/黒塗り/透明化 → BEN2
```

BEN2公式READMEでも、入力は通常の画像または動画で、 `refine_foreground` はエッジ改善の追加後処理として用意されていますが、そのぶん推論時間は増えます。動画処理では `segment_video` 、WebMアルファ出力、batch指定などが用意されています。([github.com](https://github.com/PramaLLC/BEN2))

## 裏取り：BEN2にmask入力の口は無い

公式のAPIシグネチャを一次（HF/GitHub）で確認しました。入力できるのは画像か動画パスだけです。

画像側は model.inference(image, refine\_foreground=False) [\[3\]](https://huggingface.co/PramaLLC/BEN2/blob/main/README.md) で、引数は画像と `refine_foreground` のみ。マスクやプロンプトを渡す引数はありません。

動画側もソースコードのシグネチャを直接確認しました。def segment\_video(self, video\_path, output\_path="./", fps=0, refine\_foreground=False, batch=1, print\_frames\_processed=True, webm=False, rgb\_value=(0, 255, 0)) [\[4\]](https://github.com/PramaLLC/BEN2/blob/main/BEN2.py) ——ここにもマスク誘導の引数は無い。

つまりBEN2は「画像/動画を見て前景を自前で推定する」モデルであって、Confidence Guided Matting (CGM) パイプラインを用い、ベースモデルが低信頼とした画素を専用リファイナネットワークで処理する [\[2\]](https://huggingface.co/PramaLLC/BEN2) 構造です。 **確信度を判断するのはBEN2自身の内部であって、外部マスクを正解として受け取る設計ではありません。** だからSAMの logits/probability を作っても、それを流し込む先がない、というのが核心です。

## では「2と3で作った正解マスク」をどう活かすか

ここで設計を二択に整理します。あなたの本来の目的（複数対象の個別抜き＋髪エッジ品質）から、 **現実的なのはBに寄せる構成** です。

### 案A：BEN2を画像のまま再処理（crop方式）

BEN2に渡せるのは「素の画像」なので、マスクを注入する代わりに、 **SAM/BBOXで対象領域をcropした"元画像"をBEN2に入れる** 。

```
SAM/BBOX で対象の拡張crop領域を決める
  ↓
元動画フレームの crop（加工しない素のRGB）を BEN2.inference に入れる
  ↓
crop内のBEN2アルファを取得
  ↓
SAMマスクの境界band（数px〜数十px）だけにBEN2アルファを採用
  ↓
core=不透明 / outside=透明 / band=BEN2 で最終合成
```

ポイントは前回GPT-5.5も触れた通り、 **BEN2に渡すcropは黒塗りやinpaintで加工しないこと** 。加工すると照明・テクスチャの不整合がBEN2にとって偽の前景境界に見え、品質が落ちます。マスクは「どこをcropするか」と「最後にどこへ貼り戻すか」の制約に使い、BEN2には素のピクセルだけ見せる。これが「マスクを活かす」現実解です。

### 案B：そもそもBEN2を使わず、SAMの確率マスクをマット化する

「2と3でlogits/probabilityまで持っている」なら、実は **BEN2を通さずにそれをアルファ源にする** ほうが素直なケースが多い。SAM2.1の確率はソフト値なので、二値化せずに：

```
SAM probability（ソフト）
  ↓ 境界帯のみ
guided filter / matting refinement で境界をなじませる
  ↓
final alpha
```

この場合BEN2の役割は「全体前景のプレビュー（ステップ1）」と「対象を取りこぼした時のcrop救済（案A）」に限定されます。

## どちらを選ぶかの判断軸（あなたの「軽さ」基準で）

案Bが軽い。SAM確率→guided filterはGPU負荷が小さく、対象数が増えても破綻しにくい。「回し続けても破綻しない」要件に合います。案AはBEN2をcropごとに再実行するので、対象数×フレーム数だけ推論が増える。ただし 髪のマッティング、エッジ精細化に優れる [\[3\]](https://huggingface.co/PramaLLC/BEN2/blob/main/README.md) BEN2を境界bandに限定投入できるので、髪が主役の素材では品質が効きます。 **髪が勝負なら案A、量産・速度なら案B** 、が切り分けです。

## 要件定義書 ver1.0

## 動画αマット生成パイプライン（MVP）

### 1\. 目的

追跡対象（人物等）を動画から切り抜き、 **透明背景（RGBA / α）として出力** する。本MVPの目的は処理速度の最適化ではなく、 **10秒程度のテスト動画で、出力αが主観的に満足できる品質に達するかを検証すること** 。

### 2\. スコープ

| 区分 | 内容 |
| --- | --- |
| 対象（in） | 単一カット動画のα生成、ロスト復帰、毛先エッジ品質の確認 |
| 対象外（out） | forward-only/チャンク化の実装最適化、occlusion閾値の精密チューニング、distractor乗り換え対策、リアルタイム化、長尺（>30秒）最適化 |

> 対象外は「いま詰める問題でない」とのことなので、評価フェーズを止めない論点として意図的に切り離した。品質評価がOKになった後に着手する。

### 3\. 入出力定義

| 項目 | 仕様 |
| --- | --- |
| 入力 | 動画ファイル（想定3〜5秒、テストは約10秒） |
| 中間 | 検出BBOX＋ID、SAM2二値マスク、BEN2ソフトα |
| 出力 | フレーム連番のRGBA（またはαチャンネル系列）。後段で透明背景合成／UE5受け渡しに使える形 |

### 4\. 機能要件

- **F-1 追跡対象のα生成** ：対象を毎フレームα化する。
- **F-2 ロスト復帰** ：対象を見失った段階で再追跡・再プロンプトし、ロストフレームを最小化する（ロスト＝品質劣化に直結のため最重要）。
- **F-3 毛先エッジ** ：SAM2二値では描けない毛先を、BEN2のソフトα＋膨張ゲートで補う。
- **F-4 複数対象** （任意）：複数被写体時もIDごとにαを出せること（MVPでは1名で可、余力で確認）。

### 5\. 非機能要件

| 項目 | 基準 |
| --- | --- |
| レイテンシ | 10秒動画に対しα出力まで **1〜2分以内は許容** 。「10秒に30分」級は不可。 |
| 破綻条件 | O(N²)の全再伝播でコスト爆発しないこと（設計上の禁止事項） |
| 再現性 | 同一入力で同一出力（乱数なし、もしくはseed固定） |

### 6\. 受け入れ基準（＝MVPの合否）

1. 約10秒のテスト動画で、 **ロストフレームが体感ゼロ〜許容範囲** 。
2. **毛先が「黒縁/ベタ抜き」になっていない** （BEN2ソフトαが効いている）。
3. 出力αを透明背景に合成して **主観的に満足** できる。
4. 生成完了が **2分以内** 。

> 合否は数値KPIではなく主観評価を一次基準とする（MVPの狙いが「満足いくか」の確認のため）。

## A案 仕様書 ver1.0 ―「毎フレーム検出」愚直ベースライン

### A-1. コンセプト

SAM2の動画propagation／メモリを **使わない** 。毎フレーム、検出→単画像セグメント→αを独立に回す。 **最も実装が単純で、ロストに構造的に強い** （毎フレーム取り直すので“見失う”概念がそもそも薄い）。MVPの品質ベースライン取得用。

### A-2. 処理フロー

```
各フレーム f について:
  RF-DETR(Nano) で検出  → BBOX
  ByteTrack で ID 付与（時間整合）
  SAM2 を単画像モードで BBOX プロンプト → 二値マスク
  BEN2 でソフトα生成（マスク内＋膨張ゲートで毛先）
  RGBA 出力
```

### A-3. 構成要素

| 役割 | 採用 | 備考 |
| --- | --- | --- |
| 検出 | RF-DETR Nano | 軽量、毎フレームでも破綻しにくい |
| ID | ByteTrack | フレーム間IDの一貫性 |
| マスク | SAM2（image mode） | propagation不使用、毎フレーム単発 |

### A-6. MVP適性

**◎ 最初に作るべき。** 品質の「素のベースライン」を最速で得られ、B案の良し悪しを測る物差しになる。

## B案 仕様書 ver1.0 ―「propagation＋ロスト再追跡」目標アーキ

### B-1. コンセプト

### B-2. 処理フロー


### B-4. ロスト対応（本案の核）

## ルートA 仕様書 ver1.0 ―「ブラー誘導 → BEN2再アルファ化」

### A-1. コンセプト

SAM2領域の **外側をブラー** して被写体だけをシャープに残した動画を作り、それをBEN2に通す。BEN2はsaliency（顕著性）ベースのαマット生成なので、 **「ピントが合っている＝主役」という手がかりを与えて注目を誘導する** 狙い。ハードな切り取りでなく、ブラーという連続的な勾配で背景を抑えるため、 **境界・毛先のソフトさを壊しにくい** のが期待点。

### A-2. 処理フロー

```markdown
1. SAM2.1 で対象の二値マスク M を取得
2. M を膨張 → ゲート領域 G（毛先を巻き込む余白つき）
3. 元フレーム I に対し、G の外側を強ブラー → ブラー画像 I'
   （G の内側＝被写体はシャープ維持）
4. I' を BEN2 に入力 → ソフトα を生成
5. RGBA 出力
```

### A-3. 構成要素

| 役割 | 採用 | 備考 |
| --- | --- | --- |
| 領域 | SAM2.1（二値マスク） | ブラー範囲を決める下地 |
| ゲート | 膨張マスク G | 毛先の逃げ代を確保 |
| ブラー | Gaussian等（背景側のみ） | 強度はチューニング対象 |
| α生成 | BEN2 | ブラー画像から再α化 |

### A-4. 長所／短所

| 長所 | 短所 |
| --- | --- |
| ハードカットしないので **毛先のソフトαが残りやすい** | ブラー1パス分の処理が増える |
| BEN2が背景の偽陽性を拾いにくくなる（誘導効果・ **仮説** ） | ブラー誘導が効く保証がない（saliencyが背景境界に反応する可能性） |
| 境界が自然（連続勾配） | ブラー強度・カーネルが弱いと背景残り、強すぎると境界に滲み |

### A-5. リスク（要検証）

- **誘導効果そのものが仮説** 。ブラーでBEN2のαが本当に被写体へ寄るかは実測必須。効かなければルートAの存在意義が消える。
- ブラー境界（G の縁）にBEN2が反応して **偽のエッジ** を作る恐れ。Gを十分外側に取って境界をBEN2の関心外へ追い出す設計で緩和。

## ルートB 仕様書 ver1.0 ―「SAM2.1領域前提 → BEN2αの不要部分を消す」

### B-1. コンセプト

BEN2を通常どおり全画面に走らせてソフトαを生成し、 **SAM2.1の（膨張）領域マスクでαをゲートして、領域外のαを消す** 。BEN2が他の顕著物体（別人・小物）を拾った **偽陽性を、SAM2の対象IDで刈り取る** のが目的。実装が最も単純で軽い。

### B-2. 処理フロー

```
1. SAM2.1 で対象の二値マスク M を取得 → 膨張 → ゲート G
2. BEN2 を元フレームにそのまま適用 → ソフトα A_ben
3. α合成：A_out = A_ben × softgate(G)
   （G 内はそのまま、G 外は 0 へ。境界は羽根化して段差を防ぐ）
4. RGBA 出力
```

### B-3. 構成要素

| 役割 | 採用 | 備考 |
| --- | --- | --- |
| 領域 | SAM2.1（二値マスク） | ゲートの母体 |
| ゲート | 膨張＋羽根化（feather）G | ハード段差回避が肝 |
| α生成 | BEN2（全画面） | 誘導なしの素のα |
| 合成 | α × softgate | 単純乗算ベース |

### B-4. 長所／短所

| 長所 | 短所 |
| --- | --- |
| **最軽量・最短実装** （後段の乗算1回） | ゲートが固ければ **毛先が切られる** （毛先がG外に伸びると消える） |
| BEN2の領域外偽陽性を確実に除去 | SAM2のロスト/欠けがそのままαの欠けに直結 |
| デバッグ容易（αとGを別々に見られる） | G の膨張量・羽根幅が毛先品質を左右（チューニング必須） |

### B-5. リスク（要検証）

- **毛先がG外に出ると刈られる** ＝ユーザー最重要の毛先品質と正面衝突しうる。膨張量と羽根化幅で逃がすが、過大膨張は背景偽陽性を呼び戻すトレードオフ。
- SAM2側のロストがそのまま穴になる（ルートAより穴に弱い）。
