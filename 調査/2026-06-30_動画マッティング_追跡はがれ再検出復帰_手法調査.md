## 1. 要約（結論先出し）

- **性能制約は実は緩い**: 「1分→50分以下」は 30fps 換算で **1フレーム約1.67秒** の予算。重い再検出を毎フレーム避ければ、**周期/トリガ的に GroundingDINO 再検出を挟む B案は十分現実的**（算術は §4。MAM matting 後処理の実コストは未計測）。
- **ベースは SAM2 + SAMURAI が最有力**。SAM2 は streaming memory による伝播追跡を持ち、**追跡開始後に新オブジェクトを追加可能**（公式 README 確認済）→ はがれ後の re-prompting／再シードの土台になる。SAMURAI は **再学習不要の motion-aware memory** で「はがれにくさ」を底上げ（arXiv 確認済）。**本リポジトリは既に `samurai/` を同梱**しており実装ハードルが低い。
  - **【重要・確定】SAMURAI は単一オブジェクトのみの追跡手法**（詳細・根拠は §8-1）。複数被写体を同時に扱う場合は SAMURAI ではなく **SAM2 per-object VOS** を使う。SAMURAI を複数追跡前提で設計しないこと。
- **lost からの能動復帰**は SAM2/SAMURAI 単体には明示機能が無い（未確認）ため、**外部トリガ（マスク面積急減等）＋GroundingDINO 再検出で再シードする B案を上に被せる**のが妥当。
- 比較対象の **A案（毎フレーム検出＋ByteTrack）** は MOT 実績豊富だが、**bbox 追跡でありマスク品質は別途 SAM2 が必要**。matting では SAM2 系を主軸にする方が素直（推測）。

## 2. 手法カタログ

### 2-1. ByteTrack（tracking-by-detection の基準）
出典: github.com/FoundationVision/ByteTrack（`ifzhang/ByteTrack` からリダイレクト）, arXiv 2110.06864
- **概要**: MOT を「全検出ボックスの関連付け」として解く。高スコアだけでなく**低スコア検出も tracklet との類似度で復元**し遮蔽断片化を抑える。
- **lost復帰**: 低信頼検出の救済が主。`track_buffer`（一定フレーム tracklet を保持）＋Kalman 短期 coasting。**buffer の具体フレーム数は README 未明記＝未確認**。
- **速度**: 論文値で V100 単体 30 FPS（YOLOX-X 込み）。関連付け処理自体は非常に軽い。
- **matting適合**: 出力は **bbox+ID のみ**。マスクは SAM2 等に委譲前提。
- **長所**: 速い・枯れている・任意検出器に後段接続可。 **短所**: 外見特徴を使わず長時間遮蔽後の同定が弱い（ID switch）。

### 2-2. OC-SORT（遮蔽・非線形運動に強い動き専用トラッカ）
出典: github.com/noahcao/OC_SORT, arXiv 2203.14360
- **概要**: 純動きモデル。Kalman の「遮蔽中に priori を盲信して誤差蓄積」を修正。
- **lost復帰（注目）**: **Observation-Centric Re-Update (ORU)**。遮蔽明けの再検出時に、遮蔽前観測と再出現観測を結ぶ**仮想軌跡 (virtual trajectory)** を作り、その期間の Kalman を再計算して誤差を打ち消す。→「はがれ→復帰」のクリーンな実装例。
- **速度**: RTX2080Ti でフル約28 FPS、検出済み入力なら association のみ i9 CPU 700+ FPS。
- **matting適合**: bbox 追跡。ただし virtual trajectory の考え方は B案の再シード設計に流用価値が高い（推測）。
- **長所**: 遮蔽・非線形運動に強い、ReID 不要で軽い。 **短所**: 外見を見ず、似た個体の交差で取り違えうる。

### 2-3. BoT-SORT / Deep OC-SORT / StrongSORT（ReID で再同定強化）
出典: github.com/NirAharon/BoT-SORT, arXiv 2206.14651（StrongSORT・Deep OC-SORT は README リンクのみ＝詳細未確認）
- **概要**: 動き＋**外見特徴 (ReID 埋め込み)**＋**カメラ運動補償 (CMC)**＋改良 Kalman を統合。
- **lost復帰**: 遮蔽復帰時に **ReID 類似度で同一個体を再関連付け**。動きが破綻しても外見で復帰。
- **速度**: ReID CNN を毎対象・毎フレーム回す追加コスト。具体 FPS は README 未提示（未確認）。CMC の C++ GMC は OpenCV contrib 依存で **Windows ビルドに難**（README 注意書き）。
- **matting適合**: bbox 追跡。**単一被写体なら ReID の恩恵は限定的**（推測）。
- **長所**: 長時間遮蔽・群衆で強い。 **短所**: ReID 追加コスト・実装複雑度・Windows CMC ビルド難。

### 2-4. SAM2（伝播追跡のベース／re-prompting の土台）
出典: github.com/facebookresearch/sam2, arXiv 2408.00714
- **概要**: **streaming memory** transformer。`add_new_points_or_box`→`propagate_in_video` でマスクレット伝播。
- **lost復帰（重要）**: 「lost 自動再検出」機能の記述は無し（未確認）。ただし 2024/12/11 更新で **「追跡開始後に新オブジェクトを追加できるよう前提を緩和」**（README 確認済）＝**途中フレームで box/point 再投入＝re-prompting／再シードが API レベルで可能**（B案の実装根拠）。
- **速度**: A100 で `sam2.1_hiera_large`≈39.5 FPS、`base_plus`≈64.1、`small`≈84.8、`tiny`≈91.2。`vos_optimized=True`(torch.compile) で VOS 推論を高速化。RTX4090 実測は未確認。
- **matting適合**: **マスクを直接出力**し最も相性が良い。 **短所**: fixed-window memory ゆえ速い動き/自己遮蔽で誤差伝播（SAMURAI の動機）。

### 2-5. SAMURAI（SAM2 を「はがれにくく」する motion-aware memory）
出典: github.com/yangchris11/samurai, arXiv 2411.11922
- **概要**: SAM2 を視覚追跡向けに改良。**再学習不要のゼロショット**で SAM2.1 重みをそのまま使用（FAQ 明言）。
- **「はがれにくい」理由（核心）**: ① fixed-window memory が「記憶の質」を考慮せず誤差伝播する問題を指摘。② **Kalman で位置・スケールを予測**し **motion-aware memory selection** で「動きと整合するマスク」を選別。→自己遮蔽・高速移動で誤マスクに飛ぶのを抑制。LaSOT_ext で +7.1% AUC、GOT-10k で +3.5% AO。
- **lost復帰**: Kalman は短期 coasting には効くが、**完全 lost 後の外部再検出復帰は README 明記なし（未確認）**。**streaming/webcam 入力は未対応**と明言（FAQ。ただし本件はオフライン動画処理なので影響なし）。
- **速度**: abstract で「operates in real-time」。SAM2＋軽量 Kalman で追加コスト小（推測）。RTX4090 実測未確認。
- **matting適合**: マスク出力で matting 向き。**はがれ頻度を下げ、再検出トリガの発火回数を減らせる**（推測）。
- **実務メモ**: **本リポジトリに `samurai/` が同梱済**で `INFERENCE_TRACKER_VARIANT` 切替運用の前提あり → 採用ハードルが最も低い。

### 2-6. XMem / Cutie（長期メモリ VOS）
出典: github.com/hkchengrex/XMem (arXiv 2207.07115), github.com/hkchengrex/Cutie (arXiv 2310.12982)
- **概要(XMem)**: Atkinson-Shiffrin 記憶モデル（sensory/working/long-term）。短期頑健性と長期省メモリを両立。
- **lost復帰**: 明示的「再検出」ではなく、**長期記憶に外見を圧縮保持し長時間遮蔽後に記憶マッチングで復帰**。README は「Handling long-term occlusion」「10,000フレーム超も容易」と明記。
- **速度**: 長尺でも約20 FPS、限られた GPU メモリで動作。
- **概要(Cutie)**: XMem 後継。一貫性・頑健性・速度向上、オブジェクト追加/削除対応、`max_internal_size` で処理解像度調整可。FPS 具体値は未確認。
- **matting適合**: マスク VOS で向き。ただし**初期化にマスクプロンプトが必要**でテキスト→box→mask は外部依存（GroundingDINO+SAM）。
- **長所**: **長期遮蔽復帰が明示的に強い**・省メモリ・長尺安定。 **短所**: プロンプト柔軟性は SAM2 未満。

> 注: GroundingDINO 本体・StrongSORT・Deep OC-SORT・DEVA は本調査では未取得（未確認）。

## 3. 再検出トリガ（lost 判定）の選択肢比較

> 個別のしきい値数値は本調査の取得情報に含まれず未確認（実装時チューニング要）。OC-SORT の virtual trajectory が「再出現観測で軌跡を補修する」具体例。

| トリガ指標 | 検知できる状況 | 計算コスト | 誤検知リスク | matting での使いやすさ |
|---|---|---|---|---|
| **マスク面積の急減/消失** | 画面外・全遮蔽・はがれ | ほぼゼロ | シーン縮小・遠ざかりで誤発火 | ◎ SAM2/SAMURAI 出力から即算出 |
| **前後フレーム IoU 低下** | マスクが別物に飛んだ | 低 | 高速移動で正常でも低下 | ○ 連続マスクで算出可 |
| **マスク信頼度低下** | モデル自身が不確実 | ゼロ（出力同梱） | しきい値依存 | ◎ 追加推論不要 |
| **メモリ類似度低下** | 外見が記憶と乖離 | 低〜中 | 照明変化で低下 | ○ XMem/Cutie で自然 |
| **周期的再検出（N フレーム毎）** | 取りこぼし全般の保険 | 中（検出器を定期起動） | 無駄打ち発生 | ◎ 単純・実装容易 |

- **周期的**: 単純で取りこぼしに強いが正常時も検出器を回す。予算が緩い（§4）ので数十フレーム毎なら許容（推測）。
- **信頼度/面積トリガ**: 正常時ゼロコスト、はがれた時だけ重い検出を起動。しきい値調整要。
- **推奨は併用**: ①面積急減＋信頼度低下を主トリガ、②保険として粗い周期再検出（推測）。

## 4. 「1分→50分以下」制約下での実現可能性評価

**予算計算（事実ベースの算術）**
- 1分＝60秒。30fps なら 1,800 フレーム（24fps なら 1,440）。
- 50分＝3,000秒。
- ⇒ **1フレーム当たり 3,000 / 1,800 ≈ 約1.67秒** の予算。**非常に緩い**。

**各手法の素の推論コスト**
- SAM2 large: A100 で 39.5 FPS ＝ 約0.025秒/frame（README 値）。RTX4090 も同オーダーと推測。
- SAMURAI: SAM2＋軽量 Kalman ＝ SAM2 とほぼ同等（推測）。
- GroundingDINO 1回: 本調査未取得。一般に 0.1〜0.3秒/frame オーダー（**推測、要実測**）。

**結論**
- **B案（SAM2/SAMURAI 伝播＋トリガ再検出）は余裕で制約内**（推測、MAM matting 後処理の実コストは要実測）。
- 仮に **GroundingDINO 毎フレームの A案的構成でも、検出が 1秒/frame 未満なら制約内**（推測）。**選定基準は速度より「はがれにくさ／復帰品質」**になる。

## 5. 推奨アプローチ

### 推奨: B案＝「SAMURAI 伝播 ＋ 面積/信頼度トリガ ＋ GroundingDINO 再シード」

```
[初期] GroundingDINO(text) → box → SAM2/SAMURAI init_state + add_new_points_or_box
   ↓ propagate_in_video（SAMURAI の motion-aware memory ではがれを抑制）
[毎フレーム] マスク面積 / 信頼度 / 連続IoU を監視
   ↓ しきい値割れ＝lost 判定
[復帰] GroundingDINO(same text) で再検出 → 新 box を add_new_points_or_box で再シード
   ↓ 伝播を再開
```

**根拠（事実ベース）**
1. SAM2 が**追跡開始後の新オブジェクト追加＝途中 re-prompting を公式サポート**（README 確認済）→ 再シードが API で素直に書ける。
2. SAMURAI が**再学習不要で SAM2 のはがれ頻度を下げる**（arXiv 確認済）→ トリガ発火を減らせる。**本リポジトリ同梱済**で導入容易。
3. トリガはマスク面積/信頼度で**追加推論ゼロ**（§3）。
4. 予算が緩い（§4）ので再検出コストは問題にならない。

**補強オプション**
- 再シード時の取り違え防止に、OC-SORT の virtual trajectory 的な「lost 直前位置と再出現 box の整合チェック（IoU/距離ゲーティング）」。
- 単一被写体なら ReID 不要。似た複数被写体の交差素材のみ BoT-SORT 的 ReID 埋め込みで補強（Windows CMC ビルド難・追加コスト注意）。
- 長時間完全遮蔽が多い素材は Cutie/XMem の長期メモリが「記憶マッチで自動復帰」する強み（README 確認済）→ SAM2 系の代替/併用候補。

### A案（毎フレーム検出＋ByteTrack）との比較所感
- **A案の利点**: 検出を毎フレーム行い「はがれ」概念が薄い。association は超軽量（OC-SORT 700+FPS）。
- **A案の欠点（matting 観点）**: ByteTrack/OC-SORT は **bbox+ID 止まり**。matting には box→SAM2 でマスク化が必要で、**毎フレーム SAM2 画像モードだと時間的一貫性（memory 伝播）を捨て、マスクのちらつきが増えやすい**（推測）。
- **総合**: matting では **SAM2 の伝播一貫性を活かす B案が素直**。A案は「被写体の出入りが激しく伝播で追従しきれない素材」の代替・保険（推測）。

### 8-1. SAMURAI は単一オブジェクトのみの追跡（確定・最重要注意）

> **【確定】SAMURAI は「単一オブジェクトのみ」を追跡する Visual Object Tracking（VOT）手法である。複数被写体の同時追跡には使えない。本プロジェクトでは SAMURAI を「複数追跡できる前提」で設計してはならない（今後のノイズ防止のため明記）。**

**根拠（本リポジトリの実コードで確認済・2026-06-30）【事実】**
- `samurai/scripts/demo.py` は first-frame の単一 bbox（`x,y,w,h` 1行）を読み、
  `predictor.add_new_points_or_box(state, box=bbox, frame_idx=0, obj_id=0)` で **`obj_id=0` の1対象だけ**初期化する。複数 obj_id を与える経路はデモに無い。
- 可視化色も `color = [(255, 0, 0)]` の**1色のみ**＝単一対象前提の実装。
- SAMURAI の核心（**motion-aware memory ＋ Kalman フィルタ**）は「**1つの動く対象**の bbox 位置・スケールを推定」する設計（README FAQ /論文）。複数対象を独立並列管理する仕組みではない。
- 評価ベンチも LaSOT / LaSOT-ext / GOT-10k / UAV123 / TrackingNet / OTB100 / NFS と**すべて単一対象 VOT**。

**補足（誤解防止）**
- 土台の **SAM2 本体**は per-object 独立推論・複数対象 VOS に対応するが、それは **SAM2 汎用 VOS の機能**であって **SAMURAI の motion-aware 追跡ではない**。SAMURAI 固有の「はがれにくさ」は単一対象にのみ効く。
- したがって複数被写体を扱いたい場合は **SAMURAI ではなく SAM2 の per-object VOS（`SAM2VideoPredictor`）**を使う、という整理にする。SAMURAI＝単一対象、と固定して扱う。

### 8-2. DEVA（Tracking-Anything-with-DEVA, ICCV2023, arXiv 2309.03903）詳細

**中核**: **decoupled video segmentation** =「タスク特化の image-level セグメンテーション」＋「タスク非依存の bi-directional temporal propagation」を分離【事実】。伝播モデルは一度学習すれば全タスクに汎化（動画アノテーション不要で track anything）。

- **GroundingDINO / SAM 連携（text-prompt mode）**【事実】: ①メモリからマスク伝播 → ②検出フレームなら **GroundingDINO** が text→bbox → ③ **SAM** が bbox→マスク → ④**伝播マスクと SAM マスクをマージ**。起動例 `demo_with_text.py --prompt person.hat.horse`。SAM variant 切替可（sam_hq 等）。
- **bidirectional propagation + consensus**【事実】: 小クリップ内でまず検出をマージ（in-clip consensus）、first frame は常に検出初期化。検出を毎周期マージするので**伝播が劣化しても定期検出で復元**しやすい【推測】。
- **複数オブジェクト同時追跡: 可能**【事実】。複数語 prompt で実績、`max_num_objects` で上限指定。matting で複数被写体を扱える設計。
- **新規/再出現の途中取り込み: 可能**【事実】。`detection_every` 間隔で周期検出 → 新規/再出現を拾う。`max_missed_detection_count` 超で memory から削除。**ただし出入りが激しいと memory に残り false positive 増**【事実】。
- **速度/メモリ**: FPS・GPU メモリの**公称数値は記載なし**【未確認】。定性的に伝播は対象数に線形、`--amp`/`chunk_size`/`detection_every` で調整可【事実】。RTX4090 で足りるかは要実測【推測】。
- **導入の障壁（重要）**【事実】:
  1. **SAM2 非対応**: 伝播は XMem 系の自前モデル。README/abstract に SAM2 言及なし → 標準では SAM2.1 マスク品質を使えない【事実/推測】。
  2. **ライセンス**: arXiv は **CC BY-NC-SA 4.0（非商用・継承）**【事実】。リポジトリの実コードライセンス（`LICENSE.md`）は未読【未確認】→ **商用 matting 用途なら NC 条項が障壁になり得るため要確認**。
  3. **Windows**: 「Ubuntu のみテスト、Windows は WSL2」。text/automatic デモは著者 fork の Grounded-SAM（GroundingDINO CUDA ビルド）必須、semi-online に Gurobi（無ければ PuLP・遅い）。Python3.9+/PyTorch1.12+ で **SAM2 が要求する torch>=2.5.1 より古い系列**。

| 新規/再出現の途中取り込み | ◎ `detection_every`で自動【事実】 | △ `--track_object_appearing_later`あり【事実】 | ○ トリガ自前設計【推測】 |

### 8-4. 推奨（追記後の更新）

- **DEVA の直接採用は避けるのが無難**: ①SAM2 非対応（matting のマスク品質要件と不一致）、②非商用ライセンス懸念、③Windows WSL2＋Gurobi 依存、の3点が実務障壁【事実ベース】。
- **ただし DEVA の設計は B案の最良の参考実装**: `detection_every` による定期再検出＋伝播マージ、`max_missed_detection_count` による memory 掃除は、B案のトリガ再検出ロジックにそのまま設計流用できる【推測】。
- **複数被写体を扱うなら SAMURAI は使わない**。SAMURAI は単一対象専用（§8-1 確定）なので、複数対象は **SAM2 per-object VOS（`SAM2VideoPredictor`）**で実装する【事実根拠あり】。
