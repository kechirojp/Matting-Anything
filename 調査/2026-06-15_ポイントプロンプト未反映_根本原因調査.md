# ポイントプロンプト未反映（グリーンバック出力に point mask が出ない）根本原因調査

- **作成日時**: 2026-06-15
- **対象アプリ**: 動画版 `gradio_app_sam2_transparent_BG_haystack_for_Movie.py`（port 7861）
- **対象 Component**: `pipelines/components/video_model_components.py`（`SAM2VideoPropagator` / `TransparentBGVideoExtractor`）
- **関連 Component**: `pipelines/components/ui_helpers.py`（`select_sam2_prompt`）, `pipelines/components/model_components.py`（`TransparentBGExtractor`）
- **関連エラー**: ERR041（box+point 併用で point が無視される → 修正済み）, ERR042（二値 guard の feather 化）, ERR039（横一直線切れ）
- **目的**: 「SAM2 Prompt Canvas にはポイントプロンプトが入っているのに、グリーンバック出力にはそのマスクが反映されない」現象の根本原因を、コード実態から特定する。**本調査では実装は行わない。**

> **重要**: ERR041 の修正で point は「黙殺」されなくなり、SAM2 に**確かに渡っている**。本件は「渡っていない（配線断）」問題ではなく、**渡った point の集約方法（1 オブジェクトに全点をまとめる）が原因で、複数領域に打った point が SAM2 の単一マスクで表現しきれず union から脱落する**問題である。

---

## 0. 結論サマリ（先に要点）

1. **point は SAM2 に渡っている**（ERR041 修正済み）。配線断や I/O 契約漏れではない。
2. **根本原因は `SAM2VideoPropagator.run` の point 集約設計**。box 群が存在するとき、**全 point（positive/negative すべて）を 1 個の追加オブジェクト `point_group_obj_id` にまとめて**登録している（[video_model_components.py 277-283 行](../pipelines/components/video_model_components.py#L277-L283), [316-323 行](../pipelines/components/video_model_components.py#L316-L323)）。
3. ユーザーは画面上で **複数の別物体（人物・複数のドラム・シンバル等）にまたがって positive 点を散布**している（添付の Canvas 画像参照）。SAM2 の 1 オブジェクトは**互いに離れた複数インスタンスを 1 枚のマスクで表現できない**ため、point グループのマスクは一部（最も支配的な連結領域）しか拾えず、**多くの positive 点が指した領域が union に入らない**。→ グリーンバック出力で「point の mask がすべて入っていない」。
4. **negative 点は box オブジェクトを削れない**。union は obj 間の OR なので、box で保持される領域に対して別オブジェクト（point グループ）の negative 点は効かない（OR で復活する）。これは ERR041 修正方針A（box 群と point 群をそれぞれ obj として登録し全 OR）の構造的副作用。
5. **guard / crop は「結果」を忠実に反映しているだけ**。`TransparentBGExtractor` は union_mask の外接矩形で crop し、`full_alpha * guard(union_mask)` で gate する（[model_components.py 647-663 行](../pipelines/components/model_components.py#L647-L663)）。union_mask に point 領域が入っていないので、guard が 0 にしているのではなく**最初から union_mask に存在しない**。修正は guard 側ではなく **propagator の集約側**で行うべき。
6. 座標系・フレーム同期・label(0/1) 変換は**いずれも正常**（後述 §4 で除外）。

## 2. 根本原因の詳細

### 2-1. 全 point を 1 オブジェクトに集約している（最有力・主因）

`SAM2VideoPropagator.run`（[video_model_components.py 277-283 行](../pipelines/components/video_model_components.py#L277-L283)）:

```python
point_group_obj_id: int | None = None
if boxes:
    target_object_ids = list(range(1, len(boxes) + 1))
    if points:
        point_group_obj_id = len(boxes) + 1   # 全 point を 1 obj に集約
        target_object_ids.append(point_group_obj_id)
```

登録（[316-323 行](../pipelines/components/video_model_components.py#L316-L323)）:

```python
if point_group_obj_id is not None:
    self._video_predictor.add_new_points_or_box(
        inference_state=state,
        frame_idx=prompt_frame_idx,
        obj_id=point_group_obj_id,
        points=np.asarray(points, dtype=np.float32),                 # 全点まとめて
        labels=np.asarray(labels or [1] * len(points), dtype=np.int32),
    )
```

**問題**: SAM2 の 1 オブジェクト = 1 連結インスタンスを表す前提。複数の離れた物体（人物・複数ドラム・シンバル）に positive 点を散らすと、SAM2 はそれらを「1 つの物体の証拠」として 1 枚のマスクに押し込めようとする。実際には:

- もっとも支配的な連結領域（または近接点集合）だけが拾われ、遠方の positive 点が指す領域は外れ値扱いになりやすい。
- 同一グループ内の negative 点は positive 点の領域から削る方向に働き、意図しない抑制を起こす。

結果として **point グループの寄与マスクが部分的にしか出ず、多数の point 領域が union から脱落** → 「point の mask がすべて入っていない」というユーザー観測に一致。

### 2-2. negative 点が box オブジェクトを削れない（構造的副作用）

union は `target_object_ids` の OR（[348 行](../pipelines/components/video_model_components.py#L348)）。box は obj 1..N、point は obj (N+1) で独立。box が保持する領域に対して、point グループの negative 点で穴を空けようとしても、**その領域は box obj が独立に保持するため OR で必ず復活**する。「ここは消したい」という negative 点が効かないケースが生じる（症状は逆方向だが同じ集約設計に起因）。

### 2-3. guard / crop は二次的（原因ではないが増幅しうる）

`TransparentBGExtractor.run`（[model_components.py 626-663 行](../pipelines/components/model_components.py#L626-L663)）は union_mask の外接矩形で crop し、`full_alpha * guard(union_mask)` で gate する。

- union_mask に point 領域が入っていれば、その領域も crop 範囲・guard 範囲に含まれ出力される。
- union_mask に入っていなければ crop 範囲外/guard=0 で消える。

つまり **guard は union_mask を忠実に反映しているだけ**で、point 脱落の原因ではない。ただし union_mask の外接矩形 crop のため、孤立した小領域が含まれても矩形が広がるだけで gate は二値（または soft）でその形状を保つ。修正対象は guard ではない。

---

## 3. なぜ「box は出るのに point は出ない」のか

添付グリーンバック画像で人物やドラムが出ているのは、それらが **box（obj 1..N）由来**で union に入っているため。box は各々が独立 obj なので複数インスタンスでも問題なく OR 統合される。一方 point は **全点が 1 obj** に潰されるため、box ほど素直に複数領域を表現できない。この非対称性が「box は出る／point は出ない」という観測を生む。

## 4. 除外した仮説（誤りではないことを確認）

## 5. 修正方針の候補（実装は次フェーズ・本調査では未実装）

> いずれも `SAM2VideoPropagator.run` の **point 登録部の集約方法**を変える方向。guard/UI は変更不要が原則。設定で切替できるようにするのが望ましい（ハードコード回避）。

1. **方針1: positive 点を最近傍 box に割り当てて box obj を補正する**
   - 各 positive 点を最も近い box obj に紐付け、その box の `add_new_points_or_box` に points/labels を同梱。negative 点も同じ box に紐付ければ box 内のくり抜きが効く（§2-2 解消）。
   - 利点: 既存 box ワークフローの「点で微調整」という直感に合う。union 構造は維持。
   - 注意: box が無い純 point 運用には別途対応が必要。

2. **方針2: positive 点ごとに独立 obj を割り当てる**
   - 1 positive 点 = 1 obj（近接 negative 点を従属）として登録し、全 obj を OR。
   - 利点: 複数インスタンスを素直に表現。box 不要。
   - 注意: 点数が多いと obj 数が増え計算コスト増。negative の従属ルール設計が必要。

3. **方針3: 集約方法を config で選択可能にする**
   - `config/inference_models.toml` 等に `point_grouping = "single" | "nearest_box" | "per_point"` を追加し切替。後方互換に `single`（現状）を残す。
   - 利点: 段階導入・回帰防止。ERR041 の後方互換方針と整合。

### 共通の検証要件（実装時）

- TDD（RED→GREEN）: 複数領域に散らした positive 点が、それぞれ union に寄与することを検証する RED テストを先に追加（torch stub / monkeypatch で `add_new_points_or_box` 呼び出し引数と union 結果を検査）。
- ERR035: 配線/UI を触る場合は Playwright 実行時検証を完了してから fixed 記録。
- 実素材の見た目は checkpoints+GPU 実行が必要なため、単体テスト＋UI 描画検証＋ユーザー GPU 実行で最終確認。

## 7. 結論

- **「point の受け渡しができていない」は表層の観測**で、実際は ERR041 で配線済み・SAM2 に到達している。
- **真因は `SAM2VideoPropagator` が全 point を単一オブジェクトに集約していること**。複数の離れた領域に打たれた point は SAM2 の 1 マスクで表現できず union から脱落し、グリーンバック出力に出ない。加えて negative 点が box obj を削れない構造的副作用もある。
- **修正は propagator の point 集約方法**（最近傍 box 割当 / 点ごと obj / config 切替）で行うのが筋。guard・UI・submodule は変更不要。
- 本調査では実装は行わない。次フェーズで方針選定 → RED テスト → 実装 → レビュー → Playwright 検証 → ドキュメント更新（WHITEBOARD/ERROR_LOG）の順で進める。

## 8. ユーザー提案①と評価レビュー（point は BBox と別マスク → positive 加算 / negative 減算）

### 8-1. 提案内容（ユーザー）

> ポイントプロンプトは BBox と別でマスクを作る。その後 BBox と合成。positive を BBox に足して、次に negative を BBox のマスクから引く。OR 合成だと添付画像のようなライン（破線）が出るので OR 合成はよくない。

### 8-2. 評価サマリ

| 観点 | 判定 |
|---|---|
| negative は OR では消せない／引くべき、という指摘 | **正しい**（§2-2 と一致） |
| positive と negative を非対称に扱う発想 | **正しい**（現状の一律 OR 欠陥を突く） |
| 「破線の主因は OR」という因果 | **要修正**（主因は OR ではなく early binarization） |
| 「add/subtract に変えれば線が消える」 | **不十分**（二値同士の演算では線は残る／subtract は新たな硬エッジを増やす） |
| negative 点だけで独立マスクを作って引く | **非推奨**（SAM2 の negative は object prompt 内で使う設計） |

### 8-3. 破線の真因は OR ではなく「早すぎる二値化（early binarization）」

[video_model_components.py 343-348 行](../pipelines/components/video_model_components.py#L343-L348):

```python
obj_mask = (mask_logits.detach().cpu().numpy() > 0.0).squeeze()  # ★logit を 0 で二値化
union_mask = obj_mask if union_mask is None else (union_mask | obj_mask)
```

- シンバル縁のように **logit が 0 付近で振動する細い領域**を threshold=0 で切ると 0/1 が交互に出て**破線**になる（エイリアシング）。ERR042 の「二値化に起因するエッジ」と同根。
- **add/subtract に変えても二値マスク同士で演算する限り破線は残る**。subtract はさらに **減算境界に新しい硬いライン**を増やすため逆効果になりうる。

### 8-4. negative の機構的な注意

- SAM2 の negative 点は本来「ある object の prompt の一部」として輪郭を内側から削るもの。**negative 点だけで独立マスク化すると「その点が属する物体全体」という過大マスク**が出て引きすぎる危険がある。
- ユーザーの目的（negative で BBox から削る）は妥当だが、**後段でのマスク減算ではなく「BBox の object prompt に negative 点を同梱して SAM2 内部で削る」のが正攻法**。これにより引きすぎと減算エッジを同時に回避できる。

### 8-5. 多領域 positive の主因は未解決

§2-1 の主因（全 positive 点を 1 object に集約 → 離れた複数領域を表現できない）は **合成方法を add/subtract に変えるだけでは解決しない**。positive 点が複数物体にまたがるなら **点を適切な object に割り当てる**（最近傍 box / 点ごと独立 object）必要がある。

### 8-6. 推奨統合方法（提案の意味論を活かしつつ線を消す）

1. **negative 点は対応する box（または positive 点群）の object prompt に同梱**して SAM2 内部で削る（後段減算しない）。
2. **positive 点は最近傍 box に割当 or 点ごとに独立 object**にして多領域を表現。
3. **合成は二値 OR ではなく logit/確率空間で**行い（object 統合は soft な `max` 等）、**最後にしきい値＋ feather（ERR042 の `feather_binary_mask`）を 1 回だけ**適用。

→ 提案の **「positive=足す／negative=引く」の意味論は採用**しつつ、実装を **(a) negative は object prompt 内で削る (b) 合成は soft 空間 + 末端 feather** に置き換えるのが、線を出さず目的を達成する形。要点は **「OR をやめる」より「二値化を遅らせて末端で feather」＋「negative は SAM2 の object prompt 内で削る」**。

## 9. ユーザー提案②と評価レビュー（動画深度推定でマスクを奥→手前に順次合成）

### 9-1. 提案内容（ユーザー）

> 動画深度推定モデルを実装後、マスクを深度情報に基づいて作る。例: 一番奥の BBox を深度マップから導いてから奥から順に合成していく（手前のマスクは奥のマスクを遮蔽しても構わない＝最終的に整合性がとれる）。BBox のマスクができた後でポイントプロンプトで作られたマスクを合成。深度推定モデルが軽ければ使いたい。

> **用語の明確化**: ユーザーは一貫して **「動画深度推定モデル（video depth estimation）」** を指している。これは単フレーム深度（image depth）をフレーム独立に回すものではなく、**モデル自体が時間的整合（temporal consistency）を内包**し、フレーム間で深度・前後関係が安定するもの（例: Video-Depth-Anything 系）。本レビューはこの前提で評価する。単フレーム深度との違いは §9-5 で明示する。

### 9-2. 評価サマリ

| 観点 | 判定 |
|---|---|
| 深度で前後関係を決め、奥→手前に z 順合成する発想 | **筋が良い**（painter's algorithm 相当） |
| 「手前が奥を遮蔽してよい」前提 | **正しい**（z-order の本質。最前面が勝つ） |
| **動画深度モデル**（時間整合内包）を選ぶ判断 | **妥当**（単フレーム深度のチラつきを設計段階で回避できる） |
| 破線（§8-3 の early binarization）の解消に深度が効くか | **限定的**（深度は前後関係を決めるだけ。境界の二値化問題は別途 feather が必要） |
| 本来の主因（point 集約）を深度が解決するか | **しない**（深度は合成順序の話。point→object 割当は別途必要） |

### 9-3. 良い点（賛成）

- **z-order 合成（painter's algorithm）は遮蔽整合の正攻法**。各 object のマスクに代表深度を与え、奥→手前に重ねれば「手前が奥を隠す」前後関係が自然に表現できる。OR の「全部 1 枚に潰す」より**インスタンス分離が保てる**点で §2 の主因（複数領域を 1 マスクに潰す）への**部分的な緩和**になりうる。
- **BBox マスク → 深度で z 順 → 最後に point マスクを合成**という段階分けは、責務分離（検出 / 追跡 / 深度 / 合成）として Haystack の Component 設計方針（疎結合・差し替え容易）に合う。深度推定を 1 Component として差せる。

### 9-4. 懸念点（重要）

1. **深度は「破線」を直接解決しない**。§8-3 の破線は早すぎる二値化が主因で、深度は前後関係を決めるだけ。**z 合成の各レイヤ境界でも二値化していれば線は出る**。深度導入と独立に、**末端 1 回の feather（ERR042）と logit 空間合成**は必須。
2. **時間的安定性は「動画深度モデルを選ぶこと」で設計的に担保する**。単フレーム深度をフレーム独立に回すと値が揺れ z 順が反転してチラつくが、**ユーザー提案どおり時間整合を内包した動画深度モデルを使えばこの問題は元から回避される**。object 代表深度の時間平滑化は補助的な保険であって、主たる安定化はモデル選択（動画深度）で達成する。
3. **z-order に深度の絶対値は不要**。必要なのは **object 間の相対前後だけ**。フル深度マップより「各 box 領域の代表深度（中央値など）の順位」で十分。ただし**順位の時間的安定が重要**なため、軽さを単フレーム化で稼ぐのではなく**動画深度モデルの中で軽量なもの**を選ぶ（§9-5）。
4. **「一番奥の BBox を深度から導く」必要性は限定的**。最終 alpha が「前景集合の union/合成」なら、**どれが最奥かを知らなくても z 順に重ねれば結果は同じ**。最奥特定は描画順の起点を決めるだけで、別途アルゴリズムを足す価値は薄い（順位ソートで足りる）。
5. **追跡との役割分担**。前後関係（occlusion）の追跡は本来 SAM2/SAMURAI のメモリが担う領域。深度を**もう 1 つの占有解決手段**として足すと、SAM2 の occlusion 判定と競合・二重管理になりうる。**深度は「合成順序のヒント」に限定**し、マスク生成自体は SAM2 に任せる切り分けが安全。

### 9-5. 「軽い動画深度推定モデル」の現実性（「軽ければ使いたい」への回答）

> ユーザー要望は **「動画深度推定モデルが軽ければ使いたい」**。よって候補は **単フレーム深度ではなく、時間整合を持つ動画深度モデルの中で軽量なもの** を探す。

- **動画深度モデル（時間整合内包）**: **Video-Depth-Anything**（small/large の系列あり）, **NVDS**, **ChronoDepth** 等。フレーム間の深度・前後関係が安定するため、本用途（フレームをまたぐ z 順の一貫性）に直接適合する。small 系であれば実用域に入りうる。
- **参考: 単フレーム深度**（Depth Anything V2 small / MiDaS small）は軽いが、**フレーム独立だと §9-4-2 のチラつき**が出るため本提案の主旨（動画深度）には非推奨。どうしても単フレームしか載らない場合の代替として、object 代表深度の時間平滑化で緩和する保険ルートに留める。
- **z 順位だけ欲しいので、フル高精度・絶対深度は不要**。動画深度モデルの **small 系 + box 代表深度（中央値）の順位**で軽量化する。軽さは「単フレーム化」ではなく「動画深度モデルの軽量バリアント選択」で稼ぐ。
- いずれも checkpoints・VRAM・推論速度・**時間的安定性**の実測が前提。**preflight 計測を先に取り**、許容 fps とメモリに収まるか、z 順がフレーム間で安定するかを確認してから採否を決める（SAM2/DINO の遅延調査と同じ手順）。

### 9-6. 推奨スタンス

- **z-order 合成の導入は中長期的に有望**だが、**point 未反映問題の直接解ではない**。優先度は §8 の修正（point→object 割当 + soft 合成 + 末端 feather）が先。
- 深度は **「box/object の相対前後を決める合成順序ヒント」に限定**し、Haystack の独立 Component（`DepthRanker` 等、I/O は frames → object 深度順位）として疎結合に差す設計が望ましい。マスク生成は SAM2 に残す。
- **二値化を遅らせる + 末端 feather** は深度の有無に関わらず必須。深度はその上に乗る合成順序の改善であって、エッジ品質の解決策ではない。
- 採否は **軽量な動画深度推定モデル（時間整合内包）の preflight 実測（fps / VRAM / 時間的安定性＝z 順の一貫性）** を取ってから判断する。単フレーム深度に代替しない。

### 9-7. 提案①②の関係

- **①（point の object 割当 + soft 合成 + 末端 feather）が土台**。これが無いと②を足しても破線・point 脱落は残る。
- **②（深度 z 順合成）は①の上位レイヤ**。①でインスタンス分離された各 object マスクに対し、深度で重ね順を与えるとオクルージョン整合が向上する。順序は **①を先に、②を後で**。

### 9-8. 「相対前後は現状実装から推定できるか」検証

**結論: 絶対深度は出せない。相対前後も現状の出力からは取り出せない（手がかりを捨てている＋残る手がかりも破綻する）。**

#### (a) 今は per-object 情報を捨てている

[video_model_components.py 337-352 行](../pipelines/components/video_model_components.py#L337-L352):

```python
obj_mask = (mask_logits.detach().cpu().numpy() > 0.0).squeeze()  # ① logit の大小を捨てる
union_mask = obj_mask if union_mask is None else (union_mask | obj_mask)  # ② どの obj の画素かを捨てる
```

`propagate_in_video` は **object ごと**に `out_mask_logits` を返すため、この時点では「重なり画素をどの object がより強く主張するか」という情報がある。しかし現コードは即二値化＋ OR で **(1) logit の確信度** と **(2) 画素の所有 object** を両方失う。union だけ見ても前後は復元不能。

#### (b) confidence は前後関係に使えない（実データの反例）

Detected boxes の confidence 表（ユーザー提供）:

| rank | phrase | confidence | bbox y2（下端） | 実際の前後 |
|---|---|---|---|---|
| 2 | a man | 0.525 | 3783 | ドラムセットの**中間**（手前シンバルに遮蔽される） |
| 3 | cymbal | 0.465 | 2980 | 人物より**手前** |
| 5 | cymbal | 0.348 | 3942 | 人物との前後は構図依存 |
| 7 | drum | 0.268 | 4090 | 人物より**奥**もある |

- confidence 順で「手前」と決めると **man(0.525) > cymbal(0.348)** で人間が最前面 → **実際（シンバルが人間を遮蔽）と逆転**。
- confidence は **テキスト一致度・検出確からしさ**であり、カメラ距離と無相関（`a man` は学習豊富で大きく写るため高く出るだけ）。同一 `cymbal cymbal` が 0.465 と 0.348 に割れるように、**同種でも値がばらつき順序基準にならない**。
- 前項 (a) の「logit が高い方を手前」も confidence と同じ穴（確信度 ≠ 深度）にはまる。

#### (c) bbox 接地点ヒューリスティックも破綻

bbox 下端 y2 が下＝手前、とする幾何ヒューリスティック: man y2=3783 > cymbal y2=2980 → man を手前と判定 → **やはり誤り**（man は中間）。人物が前傾してドラムセットの中に入り込む構図では接地点も破綻する。

#### (d) 含意

- **現状の検出・追跡データ（confidence / bbox 幾何 / logit）からは、「人がドラムセットの中に入り込む」構図の正しい相対前後は推定できない。**
- per-object マスクを union 前に保持すれば overlap ベースの弱い推定は可能だが、**重ならない object は順序付け不能・時間的に不安定**で、本構図には不十分。
- → **これは提案②（動画深度推定）の必要性を強める決定的な実例**。confidence も bbox 幾何も破綻する構図では、**画素単位の深度を持つモデルでしか正しい z 順は出せない**。深度は相対順位のみ使えば軽量モデルで足り、動画では時間平滑化必須、末端 feather と soft 合成は深度の有無に関わらず別途必要、という前項の結論は変わらない。
