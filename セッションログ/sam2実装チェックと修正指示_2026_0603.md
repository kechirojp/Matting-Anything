## 調査結果（ui-ux-pro-max + 根本原因診断）

### なぜ3つとも「機能せず」なのか — 根本原因は1つ

3つの UI はすべて **`prompt_frame_idx` スライダーの値**に依存しています。そして「シーク連動」は `VIDEO_SEEK_SYNC_JS`（`gr.Blocks(js=...)`）が動画プレイヤーの `seeked` イベントを拾い、**DOM 操作でスライダーの value を書き換え `input`/`change` を dispatch する**方式です。

これが Gradio 5（Svelte）では**信頼できない既知パターン**です。生 DOM の `input.value=` 書き換えでは Svelte 内部 state が更新されず、バックエンドの `.change` が発火しない／発火しても値が 0 のまま、という状態になります。結果：

| UI | 実態 |
|---|---|
| プロンプト起点フレーム位置（シーク連動）スライダー | 手ドラッグなら本来は動くが、「シーク連動」自動更新部が壊れている |
| 表示中フレーム再取得ボタン | JSがスライダー値を更新できず、常に 0 付近を再取得 → 連動しない |
| シーク位置を SAM2 に反映ボタン | 上と**同一ハンドラの重複**（完全に冗長） |

つまり **video-seek → backend を確実に伝える手段が Gradio 5 に無い**のが「機能せず」の元凶で、ボタン2つは本質的に重複です。

### ui-ux-pro-max の指針
- **Single source of truth / 冗長コントロール削減**: 同じ「フレーム選択」を3経路に分けるのは混乱の元。1つに集約すべき。
- **アクションに必ずフィードバック**: スライダー `.change` で Canvas が即更新される方式は、この原則を満たす確実な経路（現状すでに配線済みで、これは動く）。

### 提案する簡素化

「動画の純正シークバー→自動同期」は Gradio では信頼性が出せません。代わりに**1本のスライダーをフレーム・スクラバーとして単一の操作元にする**のが確実かつ最小です。

- 削除: `VIDEO_SEEK_SYNC_JS`（壊れた自動同期の元凶）
- 削除: 「表示中フレームを再取得」ボタン・「シーク位置を SAM2 に反映」ボタン（スライダー `.change` と重複）
- 残す: `prompt_frame_idx` スライダー（ラベルから「シーク連動」を外し「ドラッグで Canvas 更新」と明示）、`Prompt をクリア` ボタン
- 入力動画プレイヤーは「見て何フレーム目か確認する」用に残置

これで操作は**「スライダーをドラッグ＝スクラブ → Canvas 即更新」**の1本に集約され、3つの非機能 UI が無くなります。
