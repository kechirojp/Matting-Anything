## Q1. 動画を手動シークして「DINOが箱を作りやすいフレーム」を選ばせることは可能か？

**可能です。ただし「`gr.Video` の再生シーク位置（現在の秒数/フレーム番号）をバックエンドで取得する」ことはできません。** これは Gradio の仕様上の制約です。`gr.Video` プレイヤーの `currentTime` はブラウザ側（クライアント）だけが持っていて、サーバーの Python callback には渡ってきません。だからユーザーの推測どおり「シークはできるが何フレーム目かは取れない」が正しいです。

### 解決策：フレーム番号を「明示的な入力」にする

シーク位置を読む代わりに、**ユーザーがフレーム番号を能動的に指定する UI** を足せば、正確な frame_idx が確実に取れます。これは ERR017（座標は手入力禁止）には抵触しません。あれは bbox 座標の話で、フレーム選択は別です。

実装上の素地はすでに揃っています:

- video_model_components.py は `sample_frame_indices` で任意インデックスを抽出でき、`metadata["sampled_frame_indices"]` に**実フレーム番号**を保持しています。つまり「何フレーム目か」は内部的に追跡済みです。
- 現在の第1フレーム抽出は `max_frames=1, frame_step=1` 固定で **frame 0 決め打ち**（gradio_app_sam2_transparent_BG_haystack_for_Movie.py）。ここを「指定フレーム抽出」に拡張するだけです。

具体的な UI 案（精度・確実性の順）:

| 案 | 仕組み | frame_idx の正確さ |
|---|---|---|
| **A. フレーム番号スライダー（推奨）** | `gr.Slider(0, frame_count-1, step=1)` で番号指定 → `cv2.set(CAP_PROP_POS_FRAMES, n)` でそのフレームを抽出しプレビュー | ◎ 完全に正確 |
| **B. プレビュー＋前後送りボタン** | 「次/前フレーム」ボタンで1枚ずつ送る | ◎ 正確 |
| C. 秒数入力 | `gr.Number(秒)` × fps → frame_idx | ○ fps 誤差に注意 |

**A 案が最善**です。スライダーで番号を動かす→その瞬間のフレームを抽出表示→気に入ったフレームで「このフレームに DINO 検出」を押す、という流れにすれば、ドラムが大きく写るフレームを人間の目で選び、**正確な frame_idx を SAM2/SAMURAI に渡せます**。

> 注意（方針メモの指摘どおり）: fps 換算を挟む C 案は「Gradio 表示 fps」と「SAM2 入力 fps」のズレで座標伝播が崩れるリスクがあります。A/B 案は frame 番号を直接扱うので、この問題自体が発生しません。

## Q2. SAM2 は双方向伝播できるか？

**できます。** SAM2 の `propagate_in_video()` には `reverse` 引数があり、初期化フレームから**前方向にも後方向にも**伝播できます。

現状の実装は前方向のみです:

```python
# video_model_components.py L235 — 現在は前方のみ
for out_frame_idx, out_obj_ids, out_mask_logits in self._video_predictor.propagate_in_video(state):
```

双方向にするには、初期化フレーム `frame_idx=N` に box を登録した後、

```python
# 前方向（N → 末尾）
for ... in self._video_predictor.propagate_in_video(state, reverse=False):
    ...
# 後方向（N → 先頭）
for ... in self._video_predictor.propagate_in_video(state, reverse=True):
    ...
```

の2回回せば、**動画の途中フレームを初期化点にしても全フレームをカバー**できます。

### 重要な前提（今の設計とのギャップ）

現在の `SAM2VideoPropagator` は `add_new_points_or_box(frame_idx=0, ...)` と **frame 0 固定**です（video_model_components.py）。Q1 の「途中フレームを選ぶ」を活かすには、

1. **初期化フレーム index を可変にする**（`frame_idx=N`）
2. **N を起点に双方向 propagate する**

の2点をセットで実装する必要があります。これは方針メモのチェックリスト「初期化フレームは序盤寄りを選ぶ（双方向の保険）」を、**双方向対応すれば序盤に限定しなくてよくなる**という改善になります。

## アルファ下部が水平一直線で切れる理由（推定）

結論から言うと、**SAM2 のトラッキング自体は成功しているが、伝搬マスクの「下部」が30フレームで欠落・後退している**のが真因です。そしてそれが「水平一直線」になるのは、後段の crop 処理が原因です。

### ユーザーの2つの仮説への回答

| 仮説 | 判定 |
|------|------|
| ① SAM2 のトラッキング失敗？ | △ 部分的に正しい。完全失敗ではなく「下部領域だけマスクが後退」 |
| ② BBOX の大きさが固定のまま？ | ✗ 誤り。bbox は固定ではなく毎フレーム再計算されている |

### 仕組み（パイプラインを追った結果）

model_components.py の `TransparentBGExtractor.run` がこうなっています:

```python
bbox = mask_to_bbox(mask, padding=crop_padding, image_shape=image_rgb.shape)  # 毎フレーム再計算
x_min, y_min, x_max, y_max = bbox
crop_rgb = image_rgb[y_min:y_max, x_min:x_max]      # SAM2マスクの矩形で切り抜き
...
full_alpha = np.zeros(...)                           # 全面ゼロから
full_alpha[y_min:y_max, x_min:x_max] = alpha_crop   # crop の外はアルファ=0のまま
```

つまり:

1. SAM2 が各フレームで出したマスクから、その**外接矩形 (bbox) を毎フレーム計算**（`crop_padding=40` 付き）。
2. transparent-background はその矩形の**内側だけ**で背景除去を実行。
3. **矩形の外（下端より下）はアルファ＝0 に固定**される。

→ あるフレームで SAM2 マスクの下端が画像の途中までしか届いていないと、その `y_max + padding` の位置が **水平な切断線**として現れます。矩形の境界なので、不規則なギザギザではなく**完全な水平直線**になるわけです。

### なぜ30フレーム後にマスク下部が後退するのか

- frame 0 で **rank-1 の DINO bbox 1個だけ**を 1 obj_id として登録している（現状の `frame_idx=0` 固定・単一 obj）。
- カメラが引いて（2:45方向）構図が変わり、手前のドラム・脚など下部領域の confidence が低下。
- 手前のドラムや脚は「人」とは別オブジェクトなので、もともと追跡対象に含まれにくい。
- 結果、SAM2 マスクの下端が上方へ後退 → その下端が水平カット線になる。

### 本質的な対策

`crop_padding` を増やしても線が下がるだけの対症療法です。本筋は **複数 obj_id で「人＋ドラム＋脚」を別オブジェクトとして追跡し union する**こと（計画表の Phase B）。これにより下部のドラム/脚もマスクに含まれ、bbox 下端が画像下端まで届くようになります。

   sam2 は `pip install -e samurai/sam2/` でインストールして使う設計になっています（`setup.py` が存在）。インストール先は .venv 内なので、samurai ディレクトリの git 管理方法（通常 vs サブモジュール）は一切関係しません。

2. **ディレクトリの物理的な場所は変わっていない**  
   `git submodule add` は git の追跡方法を変えるだけで、samurai フォルダ自体は同じパスに存在し続けます。

3. **`sys.path` への直接追加も使っていない**  
   コード内に `sys.path` で samurai を直接追加している箇所はないため、影響ゼロです。
