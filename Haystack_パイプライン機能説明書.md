## 1. この書類で伝えたいこと

このプロジェクトは「写真や動画から、指定した人物・物体だけを透明背景（PNG の透過画像や連番画像）で取り出す」ソフトウェアです。

内部の処理は **Haystack（ヘイスタック）** という「部品（コンポーネント）をブロックのように繋げて処理の流れ（パイプライン）を作るライブラリ」を使って組み立てています。

組み込み系の言葉で例えると、

> 「各処理ブロックを、決まった形のコネクタ（入力・出力の型）で繋いだ信号処理チェーン」

に近いイメージです。ブロック単体はテストでき、コネクタの形が合えば別のブロック（別の AI モデル）に**丸ごと交換**できます。これが「API 化するときに楽」というポイントにつながります（詳しくは 9 章）。

## 2. 全体像（ざっくり図解）

```
                          ┌─────────────────────────┐
                          │   画面 / 操作 (Gradio)   │  ← 人が触るところ
                          └────────────┬────────────┘
                                       │ 「この画像/動画を処理して」
                                       ▼
                     ┌──────────────────────────────────┐
                     │      Haystack パイプライン         │  ← 本書の主役
                     │  （部品を順番につないだ処理の流れ）  │
                     └────────────┬─────────────────────┘
                                       │ 部品ごとに処理
                                       ▼
              ┌───────────┐   ┌───────────┐   ┌───────────┐
              │ 検出する部品 │→ │ 追跡する部品 │→ │ 切り抜く部品 │→ 保存する部品
              │(何がどこに) │   │(どこへ動くか)│   │(背景を消す)  │
              └───────────┘   └───────────┘   └───────────┘
                                       │
                                       ▼
                          ┌─────────────────────────┐
                          │  出力（透過PNG／連番画像） │
                          └─────────────────────────┘
```

**ポイント**: 「画面」「処理の流れ」「実際の AI モデル」の 3 層に分かれています。

```
 レイヤー          役割                              このリポジトリでの担当
┌───────────┐   ┌───────────────────────────┐   ┌───────────────────────────┐
│ 画面レイヤー │   │ 人の操作を受け付ける         │   │ Gradio（Webブラウザの画面）  │
├───────────┤   ├───────────────────────────┤   ├───────────────────────────┤
│ 制御レイヤー │   │ どの部品を、どの順で動かすか │   │ Haystack パイプライン        │
├───────────┤   ├───────────────────────────┤   ├───────────────────────────┤
│ 実処理レイヤー│   │ 実際の画像処理・AI 推論     │   │ SAM2 / GroundingDINO / BEN2 等│
└───────────┘   └───────────────────────────┘   └───────────────────────────┘
```

画面（Gradio）が無くなっても、制御レイヤー（パイプライン）はそのまま **API サーバーの中身として使い回せます**。これが本書のいちばん伝えたい構造です。

## 3. 用語のかんたんな説明

過剰な略語を避けるため、まず一覧で意味を揃えます。

| 用語 | かんたんな意味 |
|------|----------------|
| Haystack | 処理ブロックをつないでパイプラインを作るためのライブラリ（Python） |
| コンポーネント（部品） | 1 つの仕事だけを行う処理単位（例：「検出だけする」「保存だけする」） |
| パイプライン | 部品を順番に繋いだ、処理全体の流れ図 |
| ソケット | 部品と部品をつなぐ「入出力の口」。型（データの形）が決まっている |
| マスク | 「ここが対象物」という白黒（または濃淡）の型紙のようなデータ |
| アルファ（α） | 透明度の情報。0 が完全に透明、1 が完全に不透明 |
| 検出（Detection） | 画像の中から「対象物がどこにあるか」を四角い枠（bbox）で見つけること |
| 追跡（Tracking） | 動画の中で「対象物が次のフレームでどこに動いたか」を追い続けること |
| マット／マッティング | 背景を取り除いて対象物だけを切り抜く処理そのもの |
| ウォームアップ | 部品が実際に使われる直前に、AI モデルの重い読み込みを行うこと（起動を速くするための工夫） |

## 4. 系統A：静止画（1枚の写真）から切り抜く

### 4-1. 概要

1 枚の画像を入力し、「文章で対象を指定する」「マウスで枠を指定する」のどちらかの方法で対象を選び、その部分だけ背景を透明にして出力します。

### 4-2. 処理の流れ（図解）

実際の画面アプリでは、下図の 3 つの小さなパイプライン（と検出部品）を**画面側のコードが順番に呼び出し**、前段の結果（枠やマスク）を次段へ手渡しする形になっています。「1 本の大きなパイプラインに全部つながっている」わけではなく、「用途ごとの小さなパイプラインを画面が仲介する」構成です。

```
入力画像
   │
   ▼
┌───────────────────────────┐   文章プロンプト（例："person playing drums"）
│ GroundingDINO 検出部品        │ ◀──────────────────────────
│（単体で呼び出す。パイプライン化 │
│  していない検出だけの部品）    │
└─────────┬─────────────────┘
          │ 枠（bbox）候補
          ▼ ……ここで画面側のコードが結果を一時保存し、次のパイプラインへ渡す……
┌───────────────────────────┐
│ SAM2 用パイプライン            │
│ ImageNormalizer → SAM2Segmenter│  枠や点の情報から、対象の輪郭（マスク）を作る
└─────────┬─────────────────┘
          │ マスク候補一覧（複数対象は画面上で選んで合体できる）
          ▼ ……画面側のコードが選んだマスクを次のパイプラインへ渡す……
┌───────────────────────────┐
│ 背景除去用パイプライン          │
│ ImageNormalizer →               │
│ TransparentBGExtractor →        │  マスクを頼りに背景を消し、透明画像を作る
│ SAM2GuardFilter → OutputSaver   │  （誤検出の抑制と保存も同じパイプライン内）
└───────────────────────────┘
          │
          ▼
      出力ファイル
```

### 4-3. 使っている実処理

- `GroundingDINO`：文章から対象の枠を見つける画像理解モデル
- `SAM2`：枠や点から輪郭（マスク）を作るモデル
- `transparent-background`：背景を消して透明画像を作るモデル

### 4-4. 対応ファイル

| ファイル | 役割 |
|---------|------|
| `pipelines/sam2_tb_pipeline.py` | この系統のパイプライン組み立て |
| `pipelines/components/model_components.py` | 検出・切り抜きの部品本体 |
| `pipelines/components/common.py` | 画像の正規化・マスク合体などの共通部品 |
| `gradio_app_sam2_transparent_BG_haystack.py` | この系統を使う画面（Gradio アプリ） |

## 5. 系統B：動画から追いかけながら切り抜く

### 5-1. 概要

系統 A の「1 枚の画像」を「動画の全フレーム」に拡張したものです。最初の 1 フレームで対象を指定すると、以降のフレームでは自動的に対象を追いかけ（トラッキング）、各フレームで背景を消して、動画または連番画像として保存します。

### 5-2. 処理の流れ（図解）

```
入力動画
   │
   ▼
┌────────────────────┐
│ VideoReader          │  動画を読み込み、1コマ目の画像と動画情報を取り出す
└─────────┬──────────┘
          │ 全フレーム + 動画情報
          ▼
┌────────────────────┐     1コマ目で対象を指定（文章 or 枠 or 点）
│ SAM2VideoPropagator  │ ◀───────────────────────────────
└─────────┬──────────┘     対象をフレームごとに追いかけ、各フレームのマスクを作る
          │ 全フレーム分のマスク
          ▼
┌────────────────────┐
│ TransparentBGVideoExtractor │  各フレームで背景を消す（1枚ずつ処理してメモリ節約）
└─────────┬──────────┘
          │ 透過フレーム（動画 or 連番）
          ▼
┌────────────────────┐   ┌────────────────────┐
│ VideoWriter          │   │ FrameSequenceWriter  │
│（透過動画として保存） │   │（連番PNGとして保存） │
└────────────────────┘   └────────────────────┘
```

### 5-3. 「複数の対象を同時に追う」ときのイメージ

```
フレーム内に 対象1（人） 対象2（自転車） がいる場合：

  対象1 の確からしさマップ  ┐
                           ├─ 画素ごとに「一番濃い方を採用」→ 1枚の合成マスクにする
  対象2 の確からしさマップ  ┘

  → 対象同士が重なる部分も、境目が黒い線にならないように
    「割合（0〜1）」で滑らかに混ぜる工夫が入っています。
```

### 5-4. 対応ファイル

| ファイル | 役割 |
|---------|------|
| `pipelines/sam2_tb_video_pipeline.py` | この系統のパイプライン組み立て（用途別に 4 種類） |
| `pipelines/components/video_model_components.py` | 動画の読み込み・追跡・書き出しの部品 |
| `pipelines/components/video_common.py` | 動画向けの共通処理部品 |
| `pipelines/components/ownership_resolver.py` | 対象が重なったときの「取り分」を計算する部品 |
| `gradio_app_sam2_transparent_BG_haystack_for_Movie.py` | この系統を使う画面（Gradio アプリ） |

## 6. 系統C：切り抜き方式（アルファ生成）のハイブリッド版

### 6-1. 概要

系統 B の「対象を追いかける部分」はそのまま使い、「背景を消す部分」だけを **人物用モデル（transparent-background）** と **物体用モデル（BEN2）** の 2 つに役割分担させたものです。

これは「対象の種類によって、得意なモデルを自動で使い分ける」仕組みです。

### 6-2. 処理の流れ（図解）

```
追跡結果（対象がどこにいるか）
   │
   ▼
┌───────────────────────────┐
│ 対象は人物？ 物体？ で振り分け  │
└─────────┬─────────┬───────┘
          │人物       │物体
          ▼           ▼
 ┌─────────────────┐ ┌─────────────────┐
 │ transparent-      │ │ BEN2              │
 │ background        │ │（物体の背景除去に  │
 │（人物の背景除去に  │ │  強いモデル）       │
 │  強いモデル）       │ │                   │
 └────────┬────────┘ └────────┬────────┘
          │                    │
          └─────────┬──────────┘
                     ▼
           ┌───────────────────┐
           │ 1枚の透過フレームに合成 │
           └───────────────────┘
```

### 6-3. なぜ分けるのか（かんたんな理由）

人物と物体では、背景を消すときに「得意・不得意」があります。組み込み系で例えるなら、「温度センサーと湿度センサーで別の IC を使い、後段で 1 つのデータにまとめる」ようなイメージです。それぞれの得意分野を活かし、最後に 1 つの結果へ統合します。

### 6-4. 対応ファイル

| ファイル | 役割 |
|---------|------|
| `pipelines/route_a_deva_hybrid_video_pipeline.py` | 追跡 → ハイブリッド切り抜き → 保存、をつなぐパイプライン |
| `pipelines/components/hybrid_alpha_components.py` | 人物／物体の振り分けと合成を行う部品 |
| `pipelines/components/ben2_components.py` | 物体用モデル（BEN2）の部品 |
| `pipelines/components/deva_semi_online_tracker.py` | 対象の検出・追跡をまとめて行う部品 |
| `gradio_app_sam2_ben2_tb_hybrid_deva_for_Movie.py` | この系統を使う画面（Gradio アプリ） |

## 7. 系統D：モデルの差し替え基盤（部品交換の仕組み）

### 7-1. 概要

「どの AI モデルを使うか」を、プログラムを書き換えずに**設定ファイル（TOML）を編集するだけ**で切り替えられる仕組みです。画面のプルダウンメニューにも自動的に反映されます。

### 7-2. 処理の流れ（図解）

```
config/inference_models.toml   ← ここにモデル一覧を書く（人が編集する場所）
        │
        ▼
┌────────────────────────┐
│ model_registry.py         │  設定ファイルを読み込むだけの部品（モデル自体は読み込まない）
└─────────┬──────────────┘
          │ 選べるモデルの一覧
          ▼
┌────────────────────────┐
│ 画面のプルダウンメニュー    │  ユーザーが一覧から選ぶ
└─────────┬──────────────┘
          │ 選んだモデルの id
          ▼
┌────────────────────────┐
│ 対応する部品（コンポーネント）│  選んだ id に応じて、実際のモデルを読み込む
└────────────────────────┘
```

### 7-3. 3 つの役割（role）

| role（役割） | 意味 | 例 |
|------|------|-----|
| `detector`（検出） | 文章から対象の枠を探す | GroundingDINO |
| `tracker`（追跡） | 動画でフレーム間を追いかける | SAM2 系トラッカー |
| `background`（背景除去） | 背景を消してアルファを作る | transparent-background, BEN2 |

### 7-4. 対応ファイル

| ファイル | 役割 |
|---------|------|
| `config/inference_models.toml` | モデル一覧の設定ファイル（正本） |
| `pipelines/components/model_registry.py` | 設定ファイルを読むだけの部品 |

## 8. かんたんな使い方（呼び出し例）

Haystack パイプラインは「辞書（dict）を渡して、辞書を受け取る」形で呼び出します。組み込み系で言う「リクエスト構造体を渡して、レスポンス構造体を受け取る」関数呼び出しに近い形です。

### 8-1. 静止画系統（系統A）の呼び出し例

実際の画面アプリ（`gradio_app_sam2_transparent_BG_haystack.py`）に合わせると、「マスクを作るパイプライン」と「背景を消すパイプライン」の 2 段呼び出しになります。

```python
from pipelines.sam2_tb_pipeline import build_sam2_maskset_pipeline, build_mask_to_matte_pipeline

# 1. パイプラインを組み立てる（起動時に 1 回だけでよい）
sam2_pipeline = build_sam2_maskset_pipeline()
matte_pipeline = build_mask_to_matte_pipeline()

# 2. 画像と枠（bbox）や点を渡して、対象の輪郭（マスク）を作る
sam2_result = sam2_pipeline.run(
    {
        "image_normalizer": {"image": input_image},
        "sam2_segmenter": {"box": box, "points": points},  # 対象を指定する枠や点
    }
)
mask = sam2_result["sam2_segmenter"]["mask_set"]  # 対象の輪郭（マスク候補一覧）

# 3. マスクを渡して、背景を消す
matte_result = matte_pipeline.run(
    {
        "image_normalizer": {"image": input_image},
        "transparent_bg": {"mask": mask},               # 手順2で作ったマスク
        "sam2_guard": {"mask": mask, "enabled": True},   # マスクからはみ出た誤検出を抑える
    },
    include_outputs_from={"transparent_bg"},            # 途中結果もほしい場合に指定
)

rgba = matte_result["transparent_bg"]["rgba"]   # 透過画像（背景が消えた画像）
alpha = matte_result["transparent_bg"]["alpha"] # 透明度だけの画像
```

> 補足: 「文章プロンプトで対象を探す」場合は、上記の手順 2 の前に `GroundingDINODetector`（検出だけを行う部品）を単体で呼び出し、見つかった枠（bbox）を手順 2 の `box` に渡します。

### 8-2. 動画系統（系統B）の呼び出し例

```python
from pipelines.sam2_tb_video_pipeline import build_sam2_tb_video_pipeline

pipeline = build_sam2_tb_video_pipeline()

result = pipeline.run(
    {
        "video_reader": {"video_path": "input.mp4"},
        "sam2_video_propagator": {"box": [x1, y1, x2, y2]},  # 最初のフレームの対象の枠
    }
)

# 出力（動画ファイルパスや連番保存フォルダ）は matte という辞書にまとまって返る
matte = result["video_writer"]["matte"]
rgba_video_path = matte["rgba_video_path"]        # 透過動画（webm/mov）のパス（作らない場合は None）
rgba_sequence_dir = matte["rgba_sequence_dir"]    # 連番PNGの保存フォルダ（作らない場合は None）
```

### 8-3. 呼び出しの考え方（まとめ）

```
渡すもの: { "部品の名前": { "その部品への入力": 値, ... }, ... }
受け取るもの: { "部品の名前": { "その部品からの出力": 値, ... }, ... }
```

**これだけ覚えれば、どの系統のパイプラインも同じ呼び出し方でよい**という点が、API 化するときの一番のメリットです。

## 9. API化するときに嬉しいポイント

1. **呼び出し方が統一されている**
   静止画でも動画でも、ハイブリッド版でも、すべて「辞書を渡して辞書を受け取る」同じ形。Web API（例：REST の JSON リクエスト・レスポンス）にそのまま対応させやすい構造です。

2. **部品ごとに単体テストできる**
   1 つ 1 つの部品が「入力の型」「出力の型」を明示しているため、API のリクエスト・レスポンスの仕様（スキーマ）をそのまま部品の型から起こせます。

3. **モデルの中身を差し替えても、呼び出し側は変わらない**
   例えば「背景除去モデルを A から B に変える」といった変更は、部品の中身の交換だけで完結し、API の入出力仕様（インターフェース）は変わりません。組み込み系で言う「ドライバだけ差し替えて、上位アプリのインターフェースは変えない」設計です。

4. **重いモデルの読み込みを後回しにできる**
   起動時にはモデルを読み込まず、実際に使う直前に読み込む工夫（ウォームアップ）が入っています。API サーバーの起動を速くしたい場合に有利です。

5. **画面（Gradio）を外しても、パイプラインだけ流用できる**
   処理の中心はパイプラインにあり、画面はその「呼び出し役」に過ぎません。将来 Web API サーバーを立てる場合も、Gradio 部分だけ FastAPI 等に置き換えれば、パイプラインのコードはそのまま使えます。
